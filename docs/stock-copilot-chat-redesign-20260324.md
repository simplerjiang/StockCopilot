# Stock Copilot Chat Redesign (2026-03-24)

## 目标

把当前“右侧堆卡片”的 Stock Copilot 重构成真正类似 GitHub Copilot 的聊天式工作台。

当前版本的主要问题不是单点 bug，而是整体产品形态错误：

- 用户输入问题后看不到明确回答，只看到一堆流程卡片。
- 重要信息被拆散在 timeline、tool cards、evidence、acceptance、replay、actions 多块区域里，认知负担过高。
- 缺少真正的消息流和上下文连续性，用户不像在“对话”，更像在“阅读调试面板”。
- 工具调用、抓取内容、证据全文、最终结论之间缺少一个主叙事线。

因此本轮设计不做小修小补，而是重构为“消息流优先”的 Copilot UI。

## 设计原则

1. 先看对话，再看细节。
2. 每轮必须有最终回答，不能只停在草案或待执行状态。
3. 显示结构化思维流程摘要，但不暴露原始 chain-of-thought。
4. MCP 调用、抓取结果、证据、降级信息都要围绕当前消息组织。
5. 默认只显示最重要内容，深层细节通过折叠和抽屉展开。
6. 聊天区要像工作台，不要像报表页。

## 需要删除或降级的现有内容

以下内容不应继续作为默认主视图的大卡片平铺：

- 计划时间线整块长卡
- 工具调用卡片整块网格
- Evidence / Source 独立整块长列表
- Copilot 质量基线整块指标墙
- Replay 基线整块指标墙
- 下一步动作独立整块 chip 墙

这些内容不是没价值，而是不应该抢占主界面层级。应改为：

- 内嵌到单轮回答消息中
- 或放进右侧详情抽屉
- 或作为消息下的可展开 section

## 新页面结构

### 一、整体布局

采用三层结构：

1. 主聊天区
2. 右侧会话详情抽屉
3. 底部输入区

默认视觉重心只放在主聊天区。

### 二、主聊天区

主聊天区按时间顺序展示消息流：

- 用户消息
- Copilot 回答消息
- MCP 工具执行事件消息
- 抓取/证据摘要消息
- 最终回答消息
- 下一步建议消息

每一轮消息组成一个可折叠的 turn group。

### 三、右侧详情抽屉

右侧详情抽屉默认折叠，按需打开，显示：

- 当前会话元信息
- replay baseline
- acceptance baseline
- 当前 symbol 的质量指标
- 原始 tool payload / traceId / degraded flags
- 历史 turn 列表

也就是说，质量指标保留，但退出默认主舞台。

### 四、底部输入区

输入区固定在底部，形态类似 Copilot / ChatGPT：

- 多行输入框
- 发送按钮
- MCP 权限/模式开关
- 当前股票上下文标签
- “新建会话”入口

## 单轮消息结构

每一轮 Copilot 回复按以下顺序组织，不再拆成并列大卡片。

## 多轮思考与关闭标准

Stock Copilot 不应是“严格一问一答立刻结束”的机器人。

它应该像 GitHub Copilot 一样，允许一轮用户问题内部经历多次受控思考与工具调用，但这个过程必须是有边界的，而不是无限循环。

这里要区分两层：

### 一、会话层

会话层是用户可见的多轮对话。

- 用户可以连续追问
- Copilot 保留上下文
- 后一轮可以引用前一轮已经获取的证据、结论和工具结果

这部分没有固定“只能几轮”的限制，直到用户新建会话、切换股票或显式清空上下文为止。

### 二、单轮内部编排层

单个用户问题内部，Copilot 允许多轮受控思考，但必须设置硬上限。

当前设计建议如下：

1. 默认最多 `3` 轮内部编排循环。
2. 每轮循环包括：
	- 判断当前问题是否已足够回答
	- 如不够，则选择下一批 MCP
	- 执行 MCP
	- 吸收证据并判断是否继续
3. 每个用户 turn 默认最多：
	- `6` 次 MCP 调用
	- 其中 `external_gated` 外部搜索最多 `2` 次
4. 单轮总耗时预算建议：
	- 普通模式 `12s`
	- Pro / 深度模式 `25s`

也就是说，它不是只能“一轮脑补”，但也不能无限追工具。

### 建议的内部循环策略

#### Round 1

- 先做问题拆解
- 优先调用 Local-First MCP
- 尝试获得最小可回答证据集

#### Round 2

- 如果 Round 1 证据不够，补调缺口工具
- 例如先有 K 线但没有公告，就补新闻；先有新闻但没有结构，就补 K 线/策略

#### Round 3

- 只允许用于补关键缺口，不允许继续发散
- 若仍证据不足，就必须输出“有限回答 + 缺口说明”，不能继续拖延

## 关闭标准

所谓“关闭标准”，就是当前这个用户问题什么时候必须结束内部思考，转入最终回答。

不是“工具全跑完才结束”，而是“已经满足回答条件，或者已经达到边界，必须收口”。

### A. 正常关闭标准

满足以下条件之一，就应结束内部循环并输出最终回答：

1. 当前问题已经有足够证据支持回答。
2. 关键证据链已经闭合。
3. 再继续调工具也不会明显改变结论。

这里的“足够证据”建议至少满足：

- 有明确问题理解
- 至少有 1 到 2 个与问题直接相关的 MCP 结果
- 有可追溯 evidence 或 traceId
- 能形成一个有方向、有依据、有风险边界的最终回答

### B. 强制关闭标准

即使答案还不完美，只要出现以下任一情况，也必须停止继续调工具，输出“有限回答”：

1. 已达到 `3` 轮内部循环上限。
2. 已达到 MCP 调用次数上限。
3. 已达到单轮时间预算上限。
4. 连续两轮没有新增有效证据。
5. 剩余工具只会重复已有信息。
6. 外部搜索未获批且本地证据已穷尽。

### C. 最终回答的最低完成标准

只要关闭，就必须产出一个真正的最终回答，而不是继续停留在 `needs_tool_execution`。

最低要求：

1. 回答结论
2. 结论方向
3. 证据依据
4. 风险或不确定性
5. 下一步建议

如果证据不足，也必须明确写成：

- 当前能回答什么
- 当前不能确认什么
- 为什么不能确认
- 若要继续，需要哪个 MCP 或哪个额外来源

## 状态机建议

单轮状态建议改成以下闭环，而不是长期卡在 `needs_tool_execution`：

1. `drafting`
2. `calling_tools_round_1`
3. `calling_tools_round_2`
4. `calling_tools_round_3`
5. `finalizing_answer`
6. `done`
7. `done_with_gaps`
8. `failed`

其中：

- `done` 表示已形成 grounded final answer
- `done_with_gaps` 表示已强制收口，但答案保留证据缺口说明
- 不允许在用户可见界面长期停留在 `needs_tool_execution` 这种半完成状态

### 1. 回答头部

显示：

- 本轮问题摘要
- 当前状态：thinking / calling tools / answering / done / failed
- 当前股票 symbol
- 耗时

### 2. 思维结构摘要

这里显示的是“结构化思维摘要”，不是原始 chain-of-thought。

格式建议：

- 用户在问什么
- Copilot 准备验证什么
- 为什么先调这些 MCP
- 当前是否还缺关键证据

应以 3 到 5 条短句展现，例如：

- 先确认用户关心的是 60 日结构和近期风险点
- 优先走本地 K 线和本地公告，不先做外部搜索
- 如果技术结构和公告都没有新风险，再给出偏中性的最终判断

### 3. MCP 调用记录

采用消息内的 compact timeline：

- `StockKlineMcp` 已调用，420ms，成功
- `StockNewsMcp` 已调用，910ms，成功
- `StockSearchMcp` 未调用，原因：当前证据足够 / 未获授权

每条记录支持点击展开，查看：

- 输入参数
- 简短结果摘要
- traceId
- warning / degraded flag

### 4. 抓取与证据内容

抓取内容不直接把大段正文塞满，而是使用“简写 + 点击展开全文”。

每条证据卡展示：

- 标题
- 来源
- 发布时间
- 1 到 2 句清洗后的证据摘要
- `展开全文` 按钮

展开后显示：

- 全文或更长摘录
- readMode / readStatus
- 原始链接

### 5. 最终回答

这是当前版本最缺的部分，必须成为每轮的核心输出。

格式建议：

- 一段结论
- 一个方向判断
- 一个置信度
- 触发条件 / 失效条件 / 风险约束

例如：

- 结论：当前更像低估值防御股的震荡整理，不支持激进追高。
- 方向：中性偏多
- 触发条件：放量突破 10.20
- 失效条件：跌破 9.85 且板块资金持续流出
- 风险：公告偏中性，缺少新的强催化

### 6. 下一步动作

动作不再单独占一整块面板，而是挂在最终回答下方：

- 看 K 线细节
- 查看公告全文
- 继续外部搜索
- 起草交易计划

动作必须与本轮最终回答一致，且只有在前置状态满足时才可点击。

## 与当前实现的映射

现有能力不是全部推倒，而是重新编排：

- `StockCopilotSessionDto` / `TurnDto` 继续保留
- MCP 调用底座继续保留
- acceptance / replay 继续保留，但转入详情抽屉
- follow-up actions 继续保留，但改挂在最终回答下面
- 多 Agent 继续保留，但不再抢主聊天层；应作为“高级分析模式”或“补强回答来源”

## 重构切片建议

### R5.1 对话壳与消息流重排

- 改主界面为聊天流
- 保留输入框、turn、tool、evidence 现有数据结构
- 去掉默认大卡片平铺

### R5.2 grounded final answer 闭环

- 所有已批准工具执行完后，必须进入最终回答生成
- 去掉长期停留在 `needs_tool_execution` 的坏状态
- 把最终回答作为消息流主输出

### R5.3 MCP 事件与证据展开体验

- 工具调用、抓取内容、证据全文采用可展开结构
- 清洗 `Evidence / Source` 摘要噪音

### R5.4 会话上下文与历史

- 做真正的连续会话
- 保留历史 turn，支持从前一轮上下文继续问

### R5.5 质量与 replay 下沉到详情抽屉

- 不删除 acceptance / replay
- 但退出默认主视图，只在详情层展示

## 本轮结论

当前 GOAL-AGENT-002 的问题，不是“缺两张卡片”，而是“默认页面叙事错误”。

新的 Stock Copilot 必须首先是一个聊天工作台，然后才是 MCP 调用器、证据查看器和交易动作入口。
