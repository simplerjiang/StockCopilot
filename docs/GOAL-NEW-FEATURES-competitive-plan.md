## P1：AI 回测验证（多窗口）

### 背景与价值

当前系统已保存每次 AI 分析的完整 commander 输出（方向、置信度、目标价、止损价），也有日 K 线历史数据，但从未将二者比对——用户无法判断 AI 建议到底准不准。

本功能将自动计算"AI 预测 vs 实际走势"，按 1日/3日/5日/10日 四个时间窗口展示命中率，让用户对 AI 能力有客观认知。

### 现有基础

- `StockAgentAnalysisHistory`：已存储完整 commander ResultJson（方向/置信度/目标价/止损价）
- `KLinePointEntity`：已有日 K 线数据，可按股票和日期查询
- 历史记录有校验层，确保只保存包含完整 commander 的有效分析

### 功能设计

#### 1.1 数据模型：`BacktestResult`

新建实体，字段如下：

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | long | 主键 |
| AnalysisHistoryId | long | 关联历史分析记录 |
| Symbol | string | 股票代码 |
| AnalysisDate | DateOnly | 分析日期 |
| PredictedDirection | string | commander 预测方向（偏多/偏空/中性） |
| Confidence | int | commander 置信度（0-100） |
| TargetPrice | decimal? | 目标价 |
| StopLoss | decimal? | 止损价 |
| Window1dActual | decimal? | 1日后实际涨跌幅（%） |
| Window3dActual | decimal? | 3日后实际涨跌幅（%） |
| Window5dActual | decimal? | 5日后实际涨跌幅（%） |
| Window10dActual | decimal? | 10日后实际涨跌幅（%） |
| IsCorrect1d | bool? | 1日预测是否正确 |
| IsCorrect3d | bool? | 3日预测是否正确 |
| IsCorrect5d | bool? | 5日预测是否正确 |
| IsCorrect10d | bool? | 10日预测是否正确 |
| TargetHit | bool? | 目标价是否在 10 日内触及 |
| StopTriggered | bool? | 止损价是否在 10 日内触及 |
| CalcStatus | string | pending / calculated / insufficient_data |
| CreatedAt | DateTime | 创建时间 |
| UpdatedAt | DateTime | 最后更新时间 |

**准确性判定逻辑**：
- "偏多"：N 日后实际收益 > 0 则判为正确
- "偏空"：N 日后实际收益 < 0 则判为正确
- "中性"：实际波动在 ±2% 以内则判为正确
- 数据不足（上市不足 N 日 / 停牌）时标记为 `insufficient_data`

#### 1.2 后端服务：`BacktestService`

**核心方法**：

`RunAsync(long historyId)` — 单条回测：
1. 读取 `AnalysisHistory`，解析 commander ResultJson 提取方向/置信度/目标价/止损价
2. 取分析日之后 10 个交易日的 K 线数据
3. 按 4 个窗口分别计算实际涨跌幅
4. 判定每个窗口的预测准确性
5. 检查目标价/止损价是否在 10 日内触及
6. 存入 `BacktestResult`

`RunBatchAsync(string? symbol, DateOnly? from, DateOnly? to)` — 批量回测：
- 过滤条件：指定股票（可选）+ 日期范围（可选）
- 跳过已有 `CalcStatus = calculated` 的记录
- 顺序处理，避免并发写冲突
- 返回处理统计（总数/成功/跳过/失败）

#### 1.3 API 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/backtest/run/{historyId}` | 触发单条回测计算 |
| POST | `/api/backtest/run-batch` | 批量回测（body: symbol?, from?, to?） |
| GET | `/api/backtest/stats` | 汇总统计（?symbol=&from=&to=） |
| GET | `/api/backtest/results` | 明细列表（?symbol=&window=&page=&size=） |

`/api/backtest/stats` 响应结构：

```json
{
  "totalAnalyses": 48,
  "windows": {
    "1d": { "count": 45, "accuracy": 0.62, "insufficientData": 3 },
    "3d": { "count": 44, "accuracy": 0.68 },
    "5d": { "count": 42, "accuracy": 0.71 },
    "10d": { "count": 38, "accuracy": 0.74 }
  },
  "targetHitRate": 0.43,
  "stopTriggerRate": 0.12,
  "confidenceCorrelation": 0.38,
  "bySymbol": [...]
}
```

#### 1.4 前端页面：回测验证页

**页面路径**：新增路由 `/admin/backtest`，菜单入口在左侧导航栏

**布局**：
- 顶部筛选栏：股票代码（可清空=全部）、日期范围、时间窗口切换（1d/3d/5d/10d）
- 核心指标卡片区（4 个 KPI 卡）：
  - 已验证分析数
  - 当前窗口准确率（大字，带颜色：≥65% 绿，50-65% 黄，<50% 红）
  - 目标价命中率
  - 止损触发率
- 各窗口准确率对比图（水平柱状图，4 个窗口并排）
- 置信度 vs 准确率散点/分组图（高置信度是否确实更准）
- 明细列表（分析日期 / 预测方向 / 置信度 / 实际涨跌 / 是否正确）

#### 1.5 K 线图标注

在现有 K 线图上，对有历史分析记录的日期标注垂直线或气泡：
- 绿色：该分析在当前窗口判定为正确
- 红色：判定为错误
- 灰色：数据不足，无法判定
- 点击气泡：弹出分析摘要（方向/置信度/目标价）

### 验收标准
1. 选择有 5 条以上历史分析的股票 → 触发批量回测 → 等待计算完成
2. 回测页出现 4 个窗口的准确率，数值与手工对比误差 < 1%
3. K 线图上出现正确/错误颜色标注
4. 点击标注弹出对应分析摘要

## P2：策略问股扩展

### 背景与价值

当前 Copilot 已支持 9 种策略关键词检测（RSI/KDJ/MACD/TD/VWAP/MA/Breakout/Gap/Divergence），但缺少国内主流的缠论和波浪理论，也没有多策略交叉确认和策略教学模式。

### 现有基础

- `StockCopilotSessionService.DetectStrategies()` 关键词匹配，可直接扩展
- `TradingWorkbenchPromptTemplates.cs` 多角色 prompt 体系
- 17 个 MCP 工具：K 线、分时、技术指标信号、新闻、基本面、市场背景、联网搜索等
- 会话持久化（`StockChatSession` / `StockChatMessage`）

### 功能设计

#### 2.1 新增策略定义

**缠论（Chan Theory）**

关键词：缠论、缠师、笔、线段、中枢、背驰、买点、卖点、盘整、缠绕

分析框架：
1. 识别当前级别（1分/5分/30分/日）的笔和线段方向
2. 判断当前中枢位置和中枢震荡范围
3. 识别背驰信号（MACD 柱面积背驰、顶背驰/底背驰）
4. 给出当前的买卖点位置（一买/二买/三买 或 一卖/二卖/三卖）
5. 结合量能变化验证信号强度

**波浪理论（Elliott Wave）**

关键词：波浪、浪型、推动浪、调整浪、五浪、三浪、子浪、浪级

分析框架：
1. 识别当前处于推动浪（1/3/5）还是调整浪（A/B/C）
2. 判断当前浪的级别（分钟/小时/日/周）
3. 计算延伸目标（黄金比例 1.618/2.618）
4. 指出当前浪的关键支撑/压力位
5. 给出下一段行情的可能方向

#### 2.2 多策略收敛评分

当用户同时使用多种策略（如"用 MACD 和 RSI 联合判断"）时：

- 提取各策略信号方向（看多/看空/中性）
- 计算策略收敛度：完全一致=高确信，多数偏向=中确信，分歧=低确信
- 收敛度评分格式：

```
多策略收敛分析
MACD：看多（金叉确认）
RSI：中性（60，未超买）
MA：看多（多头排列）

收敛度：2/3 看多 → 中高确信，建议轻仓观察
```

#### 2.3 策略教学模式

触发条件：用户问题包含"什么是"、"如何看"、"怎么用"等教学性关键词

自动切换到教学 prompt：
- 用简洁语言解释策略原理
- 举当前股票实例说明（"以当前 600519 为例……"）
- 给出该策略的适用场景和局限性

#### 2.4 前端策略选择面板

在 Copilot 对话框上方增加策略标签组：
- 展示所有可用策略（11+ 种）
- 每个标签有简短描述（hover 显示）
- 点击标签自动填充到输入框（"用[策略名]分析"）
- 支持多选（自动触发多策略收敛模式）

#### 2.5 策略信号记录

每次策略分析完成后，记录到 `StrategySignalRecord`：
- 股票/时间/策略类型/信号方向/信号强度
- 可与 P1 回测联动：查看某策略的历史命中率

### 验收标准
1. 输入"用缠论分析600519" → 收到包含笔/段/中枢/背驰判断的结构化分析
2. 输入"用MACD和RSI联合分析" → 收到多策略收敛评分
3. 输入"什么是缠论" → 收到带当前股票实例的教学性回答
4. 前端策略面板正常渲染，点击标签自动填充

## P3：筹码分布分析

### 背景与价值

筹码分布帮助判断主力成本区、套牢盘压力和获利盘多寡，是国内散户最常参考的辅助指标之一。当前系统有日K量价数据，可基于换手率衰减算法本地计算，无需外部API。

### 现有基础

- `KLinePointEntity`：日K数据（Open/High/Low/Close/Volume），本地存储
- `/mcp/shareholder`：已有东方财富股东数据（股东数量、集中度）
- 前端已有 klinecharts 图表引擎，可扩展自定义图层

### 算法设计（通达信同款换手率衰减法）

**核心思路**：每日有换手率的股票，当日的筹码在价格区间上均匀分布；随时间推移，新成交的筹码逐渐替换旧筹码。

**计算步骤**：
1. 取最近 N 日（默认 250 日）的日K数据
2. 对每日 K 线，将当日换手率的筹码均匀分布在 [Low, High] 区间
3. 对历史筹码做衰减：`旧筹码 × (1 - 当日换手率)`
4. 累加每日筹码得到完整分布
5. 归一化为百分比（各价格区间筹码占总筹码的%）

**输出**：价格区间数组（每1%或5分钱为一格）及对应的筹码比例

**关键指标**：
- 平均成本价（加权平均）
- 获利盘比例（当前价以下的筹码%）
- 套牢盘比例（当前价以上的筹码%）
- 筹码集中度（90%筹码所在价格区间宽度 ÷ 平均价格）

### 功能设计

#### 3.1 后端 API

`GET /api/stocks/chip-distribution?symbol={symbol}&date={date}&days={days}`

参数：
- `symbol`：股票代码
- `date`：计算截止日期（默认=最新交易日）
- `days`：回溯天数（默认=250）

响应：

```json
{
  "symbol": "sh600519",
  "calcDate": "2026-04-05",
  "currentPrice": 1456.78,
  "avgCost": 1380.50,
  "profitRatio": 0.72,
  "trappedRatio": 0.28,
  "concentration": 0.15,
  "distribution": [
    { "priceMin": 1200.0, "priceMax": 1220.0, "chipPct": 0.032 },
    { "priceMin": 1220.0, "priceMax": 1240.0, "chipPct": 0.041 }
  ]
}
```

计算耗时：250日数据约 50-200ms，不缓存，按需计算。

#### 3.2 前端图表组件：`ChipDistributionChart.vue`

布局：横向柱状图（水平方向）
- Y 轴：价格（从低到高）
- X 轴：筹码占比（0% → 最大值）
- 当前价：横向红线标注
- 平均成本：横向黄线标注
- 颜色区分：获利盘（绿色）/ 套牢盘（红色）
- 鼠标悬停：显示该价位的筹码比例和绝对金额估算

底部信息栏：平均成本 | 获利盘 XX% | 套牢盘 XX% | 集中度 XX%

#### 3.3 集成到股票详情

在 `StockInfoTab.vue` 中，在现有标签页组（K线/分时/基本面等）中增加"筹码"标签：
- 首次点击时加载（懒加载）
- 与 K 线日期联动：K 线拖到某日，筹码自动切换到该日的分布
- 在 K 线图上叠加筹码密集区（可选，作为图层开关）

#### 3.4 喂给 AI 分析

在 Commander prompt 的上下文中补充筹码摘要：

```
【筹码信息】平均成本 1380 元，获利盘 72%，筹码集中度 15%（偏集中）
```

辅助 Commander 判断获利了结压力和支撑区间。

### 验收标准
1. 打开任意 A 股股票 → 切换到"筹码"标签 → 2 秒内加载出分布图
2. K 线拖到历史某日 → 筹码图自动更新到该日分布
3. 获利比例 + 套牢比例之和 = 100%
4. 与同花顺/东方财富筹码图对比，主峰位置偏差 < 5%

## P4：邮件推送 + 定时自动分析

### 背景与价值

用户无法持续盯盘，希望收盘后自动收到分析摘要邮件，不需要每次手动触发分析。

### 现有基础

- `.NET BackgroundService` 体系成熟（已有 6+ 个 Worker）
- `FinancialWorker` 有时间窗口调度模式可复用
- 无 SMTP 邮件服务
- 前端设置页：`AdminLlmSettings.vue`（将改名为统一"设置"页）

### 功能设计

#### 4.1 SMTP 邮件服务：`EmailNotificationService`

配置项（存入现有配置体系）：

| 字段 | 说明 |
|------|------|
| SmtpHost | 邮件服务器地址（如 smtp.qq.com） |
| SmtpPort | 端口（465/587/25） |
| UseSsl | 是否启用 SSL |
| Username | 发件账号 |
| Password | SMTP 授权码（不是登录密码） |
| DisplayName | 发件人显示名称（默认"StockCopilot"） |
| ToAddress | 收件地址 |

发送方法：`SendAnalysisReportAsync(string symbol, AnalysisReportData data)`

错误处理：SMTP 失败不影响分析主流程，只记录到错误日志。

#### 4.2 邮件模板：分析摘要邮件

HTML 邮件格式，主要内容：

```
主题：[股票名称(代码)] AI分析报告 - 2026-04-05

贵州茅台 (sh600519) 分析报告
分析时间：2026-04-05 16:03

方向：偏多    置信度：78    

目标价：1,580    止损价：1,390

核心结论：
[Commander 核心结论文本]

关键证据：
- [证据1]
- [证据2]
- [证据3]

风险提示：
- [风险1]
- [风险2]

本报告由 StockCopilot 自动生成，不构成投资建议
```

支持批量（多只股票）合并成一封邮件，顶部有汇总表格。

#### 4.3 定时分析 Worker：`ScheduledAnalysisWorker`

调度逻辑：
- 运行在主 API 进程内（`BackgroundService`）
- 每天 15:35（北京时间，A股收盘后）触发
- 判断是否为交易日（参考现有 `TradingDayChecker` 或等价逻辑）
- 取 `ActiveWatchlist` 中的所有股票逐个分析
- 分析完成后按通知配置决定是否发送邮件
- 支持开关控制（关闭后 Worker 不触发分析）

并发控制：逐个分析，不并发（避免 LLM 并发限流）。

#### 4.4 设置页改造：`AdminLlmSettings.vue` → 通用"设置"页

**改名**：
- 路由 key：`admin-llm` → `admin-settings`（保留旧路由做重定向）
- 导航菜单标签：`LLM 设置` → `设置`
- 页面内部 title：`LLM 接口设置` → `系统设置`

**新增邮件通知区块**（在现有 LLM 配置下方）：

```
SMTP 服务器    [_______________] 端口 [___]  [SSL]
发件账号       [_______________]
SMTP 授权码    [**************]             [测试发送]
收件地址       [_______________]

[ ] 开启定时分析（每个交易日 15:35 自动分析自选股）
触发条件：[全部分析完成后发送]  （或："仅在有强烈信号时发送"）

                                   [保存通知设置]
```

#### 4.5 分析完成钩子

在 `StockAgentOrchestratorService.RunAsync()` 完成并保存历史记录后：
1. 查询通知配置是否存在且已启用
2. 如触发条件为"全部"：直接发送
3. 如触发条件为"强烈信号"：仅当置信度 ≥ 80 且方向明确时发送
4. 发送失败：记录错误日志，不重试（避免重复发邮件）

### 验收标准
1. 设置页标题/菜单已改名为"设置"，旧路由可重定向
2. 保存 SMTP 配置 → 点击"测试发送" → 收到测试邮件
3. 手动分析一只股票 → 邮件通知开启时 → 收到格式正确的分析报告邮件
4. 开启定时分析 → 模拟 15:35 触发（或手动触发接口） → 自动分析 Watchlist → 发送汇总邮件

## 开发注意事项

### 回测准确性判定的边界案例
- 停牌超过 N 日：标记 `insufficient_data`，不计入准确率
- 上市不足 N 日：同上
- 历史 K 线数据缺失：降级到数据充足的最大窗口
- 不能用"预测日收盘价"作为基准，要用"分析日收盘价"（分析可能在盘中进行）

### 筹码算法的局限性
- 算法为近似模拟，不能完全等同于涨跌停/停牌情况
- 涨跌停日换手率异常时，筹码分布可能失真，需做边界保护
- 送股/分红/合股会导致价格复权，计算前需统一使用前复权价格

### 邮件安全
- SMTP 密码存储加密（使用现有配置加密体系，不明文存入数据库）
- 收件地址仅支持单个地址（MVP 阶段），不做多收件人
- 不在错误日志中记录邮件正文（可能含敏感股票信息）
