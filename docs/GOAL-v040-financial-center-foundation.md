# v0.4.0 计划书：财报中心基础落地

> 版本：v0.4.0  
> 目标类型：正式页面化 / 结果透明化 / 数据表格化

---

## 1. 版本目标

v0.4.0 只解决一个核心问题：

> **让当前财报采集能力不再藏在管理员测试页里，而是升级为正式业务页面，并且让用户看得懂“获取到了什么”。**

本版本不强求 PDF 原件内嵌对照，不强求 RAG 正式落地，也不引入新的重型 AI 基础设施。

---

## 2. 核心范围

### 2.1 正式页面 `财报中心`

将现有测试能力升级为正式页面，建议挂到主导航，而不是继续留在设置下拉中。

### 2.2 采集结果透明化

这一项是 `v0.4.0` 最核心的用户价值之一，目标是把当前“采集结束只看到数字”的体验，升级为“用户能一眼知道这次到底抓到了什么、缺了什么、靠不靠谱”。

#### 当前问题

目前页面中的采集反馈主要停留在：

- 报表数量
- 指标数量
- 分红数量
- 融资融券数量
- 主渠道
- 总耗时

这对开发排查有价值，但对用户判断结果有效性帮助不足。用户真正关心的是：

1. 抓到了哪几期财报
2. 标题看起来是否正确
3. 这次主要用了哪个来源
4. 有没有降级
5. PDF 补充是否参与
6. 哪些内容还没抓到

#### v0.4.0 需要达到的透明化效果

采集成功后，除了数量，还要显示：

- 报告期列表
- 报告标题摘要
- 来源渠道
- 是否降级
- PDF 补充是否参与

并且这些信息不能只放在一行文本里，应明确拆成用户可读区块。

#### 结果展示建议

建议把单次采集结果拆成 4 个信息块：

1. **本次采集概览**
	- 股票代码 / 名称
	- 完成时间
	- 总耗时
	- 主渠道

2. **已获取报表摘要**
	- 报告期列表，如：`2025年报 / 2025Q3 / 2025Q2`
	- 标题摘要，如：`2025年年度报告`、`2025年第三季度报告`
	- 若标题过多，显示前 3-5 条并支持展开

3. **补充数据摘要**
	- 指标期次
	- 最近分红方案
	- 最近融资融券日期
	- PDF 补充命中情况

4. **风险 / 降级提示**
	- 是否走了 datacenter / ths 降级
	- 是否结构化字段不足
	- 是否只有部分数据成功

#### 建议新增返回字段

为了支撑这个目标，`CollectionResult` 或其对外 DTO 不应只返回 count，建议补充：

- `reportPeriods`
- `reportTitles`
- `mainSourceChannel`
- `fallbackChannels`
- `pdfSupplementSummary`
- `indicatorPeriods`
- `latestDividendInfo`
- `latestMarginTradingInfo`
- `warnings`

#### 验收口径

当用户点击一次采集后，应该能够直接回答以下问题：

1. 这次抓到了哪几期财报？
2. 标题是否像正确的财报原文？
3. 主来源是哪个？有没有降级？
4. PDF 补充是否参与？
5. 哪些内容还没拿到？

### 2.3 本地财报数据表格

这一项的目标不是“再放一个表格控件”，而是把当前零散的单股票摘要和测试视图，升级为一个真正可以管理本地财报资产的正式列表页。

#### 当前问题

目前本地财报数据虽然已经存在于 `financial-data.db` 中，但用户侧缺少：

1. 跨股票查看入口
2. 跨期次查看入口
3. 来源对比入口
4. 失败 / 缺失 / 待复核筛选能力
5. 从列表快速进入详情的正式路径

这会导致用户只能：

- 去单股票页逐个点开看
- 或回到测试页看聚合日志

这两种方式都不适合形成长期使用习惯。

#### v0.4.0 需要达到的表格化能力

增加统一表格查看本地已获取财报数据，支持：

- 分页
- 排序
- 筛选
- 查看详情
- 重新采集

#### 建议表格字段

| 列名 | 含义 | 用户价值 |
|---|---|---|
| 股票代码 | symbol | 快速定位目标标的 |
| 股票名称 | 如可补齐则显示 | 降低记忆成本 |
| 报告期 | 如 `2025-12-31` | 快速识别期次 |
| 报告类型 | 年报 / 中报 / 一季报 / 三季报 | 快速判断用途 |
| 来源渠道 | emweb / datacenter / ths / pdf | 判断数据可信度与链路 |
| 结构化状态 | 成功 / 部分成功 / 失败 | 判断是否可直接使用 |
| 字段数量 | 三表字段提取数量 | 判断完整度 |
| PDF 状态 | 无 / 已下载 / 已解析 / 失败 | 判断是否可进入后续 PDF 详情 |
| 最近更新时间 | 最近写入时间 | 判断新鲜度 |
| 操作 | 查看详情 / 重新采集 | 形成正式工作流入口 |

#### 建议筛选条件

至少应支持：

1. 股票代码 / 名称关键词
2. 报告类型
3. 来源渠道
4. 结构化状态
5. PDF 状态
6. 报告期范围
7. 最近更新时间范围
8. “只看最近一期”开关
9. “只看异常 / 待复核”开关

#### 建议排序规则

默认排序建议为：

1. 最近更新时间倒序
2. 同一股票下报告期倒序

并允许用户切换为：

- 报告期倒序
- 股票代码正序
- 来源渠道
- 结构化状态

#### 详情入口的最低要求

点击表格行后，应至少能进入详情抽屉或侧边栏，看到：

1. 报表基础信息
2. 本次采集摘要
3. 结构化结果概览
4. 来源渠道与更新时间

PDF 原件预览与重新解析可以留给 `v0.4.1`，但 `v0.4.0` 必须至少打通“列表 -> 详情”的正式路径。

#### 需要的后端接口支撑

这意味着后端不能继续只提供单 symbol 摘要接口，而需要正式支持：

- `GET /api/stocks/financial/reports`
- `GET /api/stocks/financial/reports/{id}`
- `GET /api/stocks/financial/collection-summary/{symbol}`

并支持分页、筛选、排序参数。

#### 验收口径

当 `v0.4.0` 完成后，用户应该可以通过表格快速回答：

1. 当前本地到底有哪些财报？
2. 哪些股票抓过、哪些还没抓？
3. 哪些报告期缺失？
4. 哪些来源质量较差或存在降级？
5. 哪些记录值得进一步打开详情复核？

### 2.4 正式查询接口

新增主 API 接口，用于支撑正式页面，而不是继续让前端拼装旧接口：

- `GET /api/stocks/financial/reports`
- `GET /api/stocks/financial/reports/{id}`
- `GET /api/stocks/financial/collection-summary/{symbol}`（命名可调整）

---

## 3. 本版本不做

1. 不做 PDF 原件内嵌预览
2. 不做手动重新解析
3. 不做 RAG 正式检索
4. 不做 embedding 或向量库

这些放到后续小版本。

## 4. 前端任务拆分

### 4.1 页面级

- 新建 `FinancialCenterPage.vue`
- 将 `FinancialDataTestPanel.vue` 拆分/吸收为正式页面子模块

### 4.2 组件级

建议拆分：

- `FinancialCenterToolbar.vue`
- `FinancialReportsTable.vue`
- `FinancialCollectionStatusBar.vue`
- `FinancialReportDetailDrawer.vue`

### 4.3 交互级

- 支持单股票即时采集
- 支持筛选来源/报告期/状态
- 支持从表格进入详情抽屉
- 支持采集结果卡片化展示，而不是仅显示统计数字
- 支持从采集结果区直接跳转到对应列表记录或详情抽屉

---

## 5. 后端任务拆分

### 5.1 查询接口

新增财报列表接口，支持：

- 股票代码关键词
- 报告类型
- 来源渠道
- 是否解析成功
- 报告期范围
- 最近更新时间范围

并补充分页、排序与详情查询模型，避免前端继续依赖底层读库结构。

### 5.2 采集结果增强

增强 `CollectionResult`，补充：

- `reportPeriods`
- `reportTitles`
- `pdfSupplementSummary`
- `indicatorPeriods`
- `latestDividendInfo`
- `latestMarginTradingInfo`
- `warnings`
- `fallbackChannels`
- `mainSourceChannel`

### 5.3 DTO 整理

将当前读库直返的松散对象收敛为正式 DTO，避免后续前端依赖底层字段名。

---

## 6. 验收标准

1. 用户可以从正式业务导航进入 `财报中心`
2. 用户点击采集后，能明确知道报告期、标题、来源、降级与补充情况
3. 用户可以在统一表格中查看本地已有财报，并按来源/期次/状态筛选
4. 用户可以从表格进入详情抽屉查看记录基础信息
5. 页面默认先呈现结果，不以日志为主

---

## 7. 风险

1. 若本版本尝试同时做 PDF 原件与重解析，会导致 scope 膨胀
2. 若仍然沿用测试页语言，会让页面看起来像开发工具
3. 若接口仍然沿用过于底层的读库返回，会拖累后续 `v0.4.1` 和 `v0.4.2`

---

## 8. 完成定义

当以下四点全部满足时，v0.4.0 可视为完成：

- 有正式 `财报中心`
- 有本地财报分页表格、筛选、排序与详情入口
- 有采集结果透明化，且不再只显示数量
- 有稳定的正式查询接口
