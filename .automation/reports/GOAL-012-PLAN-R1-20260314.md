# GOAL-012-R1 Chart Terminal Extensibility Upgrade Plan (2026-03-14)

## EN
### Summary
Created a parallel frontend-only task for Dev2 so chart-terminal evolution can proceed without blocking or interfering with Dev1's Step 4.2 trading-plan implementation. The new workstream focuses on evaluating a more extensible TradingView-vue or TradingView-style Vue charting solution, moving minute-line rendering into the same tab group as K-line timeframes, and preparing the chart layer for future technical-strategy overlays such as TD Sequential and KDJ crossover markers.

### Scope
- Dev1 remains on the Step 4.2 mainline: TradingPlan entity, backend APIs, draft generation, plan modal, Pending save flow, and ActiveWatchlist linkage.
- Dev2 owns the chart-terminal upgrade in the frontend layer.
- Preferred file scope for Dev2:
  - `frontend/src/modules/stocks/StockCharts.vue`
  - new files under `frontend/src/modules/stocks/charting/**`
  - `frontend/package.json` when dependency changes are required
- Dev2 should avoid touching backend contracts and should avoid modifying the Step 4.2 trading-plan flow unless a backwards-compatible chart mount seam is absolutely necessary.

### Goals
1. Evaluate and preferably adopt a highly extensible TradingView-vue or TradingView-style Vue charting solution.
2. Replace the current split K-line vs minute-line presentation with a unified tab switch:
   - `分时图`
   - `日K图`
   - `月K图`
   - `年K图`
3. Keep data compatibility with the current backend shape (`minuteLines`, `kLines`) so Dev2 does not need backend API changes.
4. Introduce a chart abstraction or adapter layer that can later host overlays like TD Sequential, KDJ crossover, additional indicators, and future annotations.

### Selection Requirements
- If TradingView-vue is viable and materially better for extensibility, use it.
- If it is not viable due to Vue 3 compatibility, licensing, maintainability, or integration limits, Dev2 must explicitly document the comparison result and justify staying on `lightweight-charts` plus an internal adapter layer.
- The success criterion is not “switch libraries”, but “gain a more extensible, professional chart interaction model”.

### Acceptance
1. Minute-line view is moved into the same chart tab switch as K-line views.
2. The visible chart buttons become `分时图 / 日K图 / 月K图 / 年K图`.
3. The chart layer exposes clear extension points for future strategy overlays.
4. Dev2 changes do not break Dev1's Step 4.2 work.
5. Frontend unit tests, build, and browser interaction validation pass.

## ZH
### 摘要
已为 Dev2 创建一条独立的前端并行任务线，使图表终端升级可以在不阻塞、也不干扰 Dev1 的 Step 4.2 交易计划开发的前提下推进。该任务的重点是评估更高扩展性的 TradingView-vue 或 TradingView 风格 Vue 图表方案，把分时图并入与 K 线周期相同的 Tab 切换，并为未来神奇九转、KDJ 金叉等技术策略叠加提供图层扩展基础。

### 范围
- Dev1 继续负责 Step 4.2 主线：TradingPlan 实体、后端 API、草稿生成、计划弹窗、Pending 保存流和 ActiveWatchlist 联动。
- Dev2 负责图表终端升级，且以前端层为主。
- Dev2 优先改动范围：
  - `frontend/src/modules/stocks/StockCharts.vue`
  - 新增 `frontend/src/modules/stocks/charting/**`
  - 如需依赖变更则修改 `frontend/package.json`
- Dev2 应避免触碰后端 contract，也应避免进入 Step 4.2 的交易计划状态流；若确需接触集成点，必须保持向后兼容。

### 目标
1. 调研并优先采用高扩展性的 TradingView-vue 或 TradingView 风格 Vue 图表组件方案。
2. 用统一 Tab 取代当前分时图与 K 线图分离展示：
   - `分时图`
   - `日K图`
   - `月K图`
   - `年K图`
3. 与现有后端数据结构保持兼容，不要求 Dev2 改动后端接口。
4. 建立图表抽象层或适配层，为后续神奇九转、KDJ 金叉、更多指标叠加和策略标记预留接口。

### 选型要求
- 若 TradingView-vue 可用且在扩展性上明显优于现方案，应优先采用。
- 若因 Vue 3 兼容、许可证、可维护性或集成限制导致不可行，Dev2 必须在回执中明确说明对比结论，并给出保留 `lightweight-charts + 自建适配层` 的理由。
- 成功标准不是“换库本身”，而是“得到更专业、可扩展的图表交互模型”。

### 验收
1. 分时图被纳入与 K 线同一组 Tab 切换。
2. 可见按钮统一为 `分时图 / 日K图 / 月K图 / 年K图`。
3. 图表层暴露清晰扩展点，可继续叠加未来策略图层。
4. Dev2 改动不破坏 Dev1 的 Step 4.2 主线开发。
5. 前端单测、构建与浏览器交互验证通过。
