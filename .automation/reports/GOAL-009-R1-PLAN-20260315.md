# GOAL-009 R1 Planning Report (2026-03-15)

## EN
### Summary
Planned GOAL-009 as a local-first market-sentiment and sector-rotation console rather than another AI-only narrative page. The first delivery slice is GOAL-009-R1: build a paged sector board backed by persisted market-sentiment snapshots, sector-rotation snapshots, and sector leader ladders so the frontend can render stable pagination, rolling comparisons, and board-detail drill-downs without scraping live endpoints on every view refresh.

### Existing Reusable Foundation
- Existing tables already cover adjacent inputs but not structured rotation snapshots: `LocalSectorReports`, `LocalStockNews`, `MarketIndexSnapshots`, `StockQuoteSnapshots`, `ActiveWatchlists`.
- Existing backend already has reusable worker patterns: `StockSyncWorker`, `HighFrequencyQuoteService`, `LocalFactIngestionWorker`, `TradingPlanTriggerWorker`, `TradingPlanReviewWorker`.
- Existing frontend already has top-level tab routing in `frontend/src/App.vue`, paged archive UI in `frontend/src/modules/stocks/NewsArchiveTab.vue`, and stock-terminal news panels in `frontend/src/modules/stocks/StockInfoTab.vue`.

### R1 Scope
1. Add a new top-level frontend tab for market sentiment and sector rotation.
2. Persist market-sentiment snapshots instead of computing everything from live HTTP responses on each page load.
3. Persist paged sector-rotation rows by snapshot time and board type.
4. Persist leader ladders for the top stocks inside each sector snapshot.
5. Expose backend APIs for latest market summary, paged sector list, sector detail, and short history windows.
6. Reuse `LocalSectorReports` only for sector-news context; do not overload it with structured ranking snapshots.

### Why New Tables Are Required
- `LocalSectorReports` is news-oriented and keyed by article-like records; it cannot support stable pagination, rolling ranking windows, or sector-score aggregation.
- `MarketIndexSnapshots` only stores one index row at a time and does not contain breadth, board concentration, or limit-pool metrics.
- GOAL-009 needs replayable, time-bucketed structured snapshots for both UI and later strategy integration.

### Planned Backend Services And Jobs
1. `ISectorRotationIngestionService`
   - crawl sector list, sector leaders, and limit pools
   - calculate sentiment and board scores
   - write snapshot tables additively
2. `SectorRotationWorker`
   - gated by `ChinaAStockMarketClock`
   - intraday cadence: every 120 seconds during trading session
   - post-close consolidation: one close snapshot after 15:05
3. `ISectorRotationQueryService`
   - latest market summary
   - paged sector list
   - sector detail + leader ladder
   - 5/10/20-day comparison window

### Issues / Risks Already Identified
- Eastmoney `getTopicQSPool` is not the broken-board endpoint and should not be used for 炸板率; use `getTopicZBPool` instead.
- Limit-pool endpoints are sensitive to sort parameters; for `getTopicDTPool`, `sort=zdp:asc` is required to retrieve sample rows reliably.
- The first slice should avoid mixing live HTTP pulls directly into the frontend, or the page will become unstable and uncached.

### 为什么必须新增表
- `LocalSectorReports` 是新闻表，按资讯文章组织，不适合做稳定分页、历史排名窗口和板块分数聚合。

- GOAL-009 需要“可回放、可分时桶聚合”的结构化快照，后面还要给策略与风控层复用。

- 东方财富 `getTopicQSPool` 不是炸板池，不能拿来算炸板率；炸板率要用 `getTopicZBPool`。
