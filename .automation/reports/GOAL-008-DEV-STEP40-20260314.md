### Development

### Follow-up Fix: recent-history switching freeze
- After Step 4.0 validation, a new runtime issue appeared: repeatedly switching recently queried stocks could freeze the page without backend exceptions.
- Root cause: `/api/stocks/detail/cache` still returned all stored minute-line rows for the symbol across multiple dates, and its K-line query returned the oldest window instead of the latest one.
- Fix:
  - Added `StockDetailCacheQueries` to centralize the cache-window logic.
  - Changed cached K-line loading to select the latest `count` rows and then sort ascending for chart rendering.
  - Changed cached minute-line loading to return only the latest trading date, capped to a single intraday session window.
  - Added backend regression tests covering both behaviors.
- Validation:
  - `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj --filter StockDetailCacheQueriesTests` -> passed (2/2)
  - `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj` -> passed (96/96)
  - Runtime check against `http://127.0.0.1:5132/api/stocks/detail/cache?symbol=sh600000&interval=day&count=60` confirmed `minuteLines` now stays on a single date (`2026-03-13`) with 241 points.

### Follow-up Fix: loading lock after third stock switch
- A second runtime issue remained after the cache-window fix: after the cached detail rendered, the UI could still stay in `loading` because the live-detail refresh had not finished, which kept the query button disabled and made the stock terminal feel frozen.
- Fix:
  - Kept the existing request cancellation logic, but split the UI semantics between blocking initial load and background live refresh.
  - The query button is now blocked only when there is no detail on screen yet; once cached detail is visible, the live request only shows `后台刷新中...` and users can continue switching symbols.
  - Added frontend regression coverage to verify that recent-history/cache-first switching stays interactive while the live refresh is still pending.
- Validation:
  - `npm --prefix frontend run test:unit -- --run src/modules/stocks/StockInfoTab.spec.js` -> passed (28/28)
  - `npm --prefix frontend run build` -> passed
  - Browser check against `http://127.0.0.1:5133/` with three consecutive symbol switches (`600000 -> 600519 -> 000001`) confirmed final symbol update succeeded and the query button remained enabled (`queryDisabled: false`).

### Follow-up Fix: preserve in-progress work per stock
- The next UX gap was semantic: switching stocks still replaced a single global state tree, so users could lose visibility into in-progress Agent/news/chat work when they switched away and came back.
- Fix:
  - Refactored `StockInfoTab.vue` to keep a dedicated workspace per symbol instead of a single shared detail/task state.
  - Bound detail, loading, agent results, news impact, local-news buckets, chat session selection, and agent-history selection to the active symbol workspace.
  - Changed stock switching semantics to behave like hiding/showing tabs: switching away no longer discards in-flight work for another symbol, and switching back restores the same workspace state.
  - Added a regression test proving an in-progress Agent run for stock A keeps running while viewing stock B, and its results are still visible after switching back to stock A.

### 后续修复：最近查询连续切换卡死
- 在 Step 4.0 验证之后，又出现了一个运行时问题：连续切换“最近查询”的股票几次后，页面会卡住，但后端没有明显异常。
- 根因：`/api/stocks/detail/cache` 仍会把该股票跨多个交易日积累的全部分时数据一次性返回，同时 K 线查询拿到的是最老窗口而不是最近窗口。
- 修复：
  - 新增 `StockDetailCacheQueries`，集中管理缓存窗口逻辑。
  - 缓存 K 线改为先取最近 `count` 条，再按升序返回给图表。
  - 缓存分时改为只返回最新一个交易日的数据，并限制在单个交易时段窗口内。
  - 补充后端回归单测，锁住这两个行为。
- 验证：
  - `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj --filter StockDetailCacheQueriesTests` -> 通过（2/2）
  - `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj` -> 通过（96/96）
  - 运行时检查 `http://127.0.0.1:5132/api/stocks/detail/cache?symbol=sh600000&interval=day&count=60`，确认 `minuteLines` 只保留单日（`2026-03-13`）且数量为 241。
