# GOAL-008 Step 4.0 Planning Report (2026-03-14)

## EN
### Summary
Planned Step 4.0 as a strict precondition slice before the actual trading-plan engine: remove visible lag when switching stocks, make charts/details appear immediately from cache, and stop stale network responses from corrupting the currently selected symbol.

### Planned Scope
- Trace the stock-terminal request/render path around `StockInfoTab.vue`, `TerminalView.vue`, chart rendering, and backend detail aggregation.
- Remove serial backend composition in `/api/stocks/detail` where quote, kline, minute, and intraday messages were awaited one by one.
- Use the existing `/api/stocks/detail/cache` endpoint as the first-read path so the terminal can render cached detail immediately before the live refresh completes.
- Add stale-response protection in the frontend so fast symbol switching cannot let an older request overwrite the newer selection.
- Remove stock terminal hardcoded `provider: 'openai'` payloads so agent/history requests respect the active backend provider switch.
- Add unit coverage for cache-first rendering and stale-response suppression.

### Planned Validation
1. `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj`
2. `npm --prefix frontend run test:unit -- --run src/modules/stocks/StockInfoTab.spec.js`
3. `npm --prefix frontend run build`
4. Browser verification against the backend-served frontend to confirm stock switching renders and interactions remain usable.

### Risks Noted
- The cache endpoint already existed, so the safer path was to reuse it rather than invent another fast path.
- Any frontend fix that updates state asynchronously needed a deterministic anti-stale guard; otherwise fast switching could regress correctness while improving perceived speed.
- Stock agent requests were still forcing `openai`, which would silently bypass the recent active-provider routing work if left unchanged.

## ZH
### 摘要
Step 4.0 被规划为真正进入“交易计划引擎”之前的前置排雷任务：先解决股票切换时的明显卡顿，让图表与详情先从缓存秒开，再补最新数据，同时保证旧请求不会把当前标的覆盖掉。

### 规划范围
- 排查 `StockInfoTab.vue`、`TerminalView.vue`、图表渲染与后端 detail 聚合路径的请求/渲染链路。
- 去掉 `/api/stocks/detail` 中 quote、kline、minute、盘中消息串行等待的后端瓶颈。
- 复用现有 `/api/stocks/detail/cache` 作为前端首读路径，让终端先显示缓存详情，再等待实时刷新完成。
- 前端加入防陈旧响应覆盖机制，保证快速切股时旧请求不会反写当前股票。
- 清理股票终端里硬编码的 `provider: 'openai'`，让 agent/history 请求遵循后端 active provider 切换。
- 增补单测，覆盖“缓存先显”和“旧响应抑制”两个关键行为。

### 计划校验
1. `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj`
2. `npm --prefix frontend run test:unit -- --run src/modules/stocks/StockInfoTab.spec.js`
3. `npm --prefix frontend run build`
4. 通过后端托管前端做一次浏览器交互验证，确认切股渲染和交互仍然正常。

### 风险记录
- 缓存接口本来就存在，最稳妥的方案是复用它，而不是再发明一套新的快路径。
- 前端异步更新若没有确定性的防串写保护，虽然体感更快，但会引入“切到新票又被旧票回包覆盖”的正确性回退。
- 股票 agent 请求此前仍强制写死 `openai`，如果不一并清理，会暗中绕过刚完成的 active-provider 路由能力。
