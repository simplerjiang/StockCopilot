# GOAL-008 Step 4.2 Planning Report (2026-03-14)

## EN
### Summary
Planned Step 4.2 as the minimal human-in-the-loop trading-plan loop built on top of the existing stock multi-agent workflow. The core design is: reuse the normalized commander output already produced by `StockAgentOrchestrator`, generate an authoritative draft on the backend from `StockAgentAnalysisHistory`, allow the user to edit it in the stock terminal UI, save it as a `Pending` `TradingPlan`, and automatically upsert the symbol into `ActiveWatchlist`.

### Current Reusable Foundation
- `frontend/src/modules/stocks/StockAgentPanels.vue` already renders commander summary, narrative rows, trigger tags, invalidation tags, and risk-limit tags. This is the right place to add the draft-plan CTA.
- `frontend/src/modules/stocks/StockInfoTab.vue` already owns stock workspace state, saved `selectedAgentHistoryId`, and an existing modal/dialog UI pattern that can host the draft-plan form.
- `backend/SimplerJiangAiAgent.Api/Modules/Stocks/StocksModule.cs` already exposes minimal APIs for agent history and watchlist operations, so Step 4.2 should follow the same style with `/api/stocks/plans` and `/api/stocks/plans/draft`.
- `backend/SimplerJiangAiAgent.Api/Data/Entities/StockAgentAnalysisHistory.cs` already persists multi-agent output and is the right backend source of truth for plan drafting.
- `backend/SimplerJiangAiAgent.Api/Infrastructure/Jobs/IActiveWatchlistService.cs` already provides the post-save watchlist integration point via `UpsertAsync`.

### Missing Functions To Add
1. Add a `TradingPlan` entity, enums, `DbSet`, migration, and schema initializer update.
2. Add `ITradingPlanService` for create/update/list/cancel operations.
3. Add `ITradingPlanDraftService` to build a deterministic draft from a saved analysis history entry.
4. Add stock-plan DTOs and minimal APIs.
5. Add a draft-plan button in `StockAgentPanels.vue`.
6. Add a draft/edit modal plus a current-plan list in `StockInfoTab.vue`.
7. Add backend and frontend tests for draft mapping, save flow, and watchlist auto-upsert.

### Planned Backend Design
- Entity fields: `Symbol`, `Name`, `Direction`, `Status`, `TriggerPrice`, `InvalidPrice`, `StopLossPrice`, `ExpectedCatalyst`, `InvalidConditions`, `RiskLimits`, `AnalysisSummary`, `AnalysisHistoryId`, `SourceAgent`, `UserNote`, timestamps, and future trigger/cancel timestamps.
- Draft generation must not call the LLM again. It should map deterministic commander fields only:
  - `AnalysisSummary` <- `analysis_opinion` or fallback `summary`
  - `ExpectedCatalyst` <- `trigger_conditions`
  - `InvalidConditions` <- `invalid_conditions`
  - `RiskLimits` <- joined `riskLimits[]` or fallback `risk_warning`
  - `TriggerPrice` <- numeric breakout-style value only when deterministically present
  - `InvalidPrice` <- numeric support-style value only when deterministically present
- Save flow must call `IActiveWatchlistService.UpsertAsync(symbol, name, "trading-plan", $"plan:{id}", true)` after a plan is created.

### Planned Frontend Flow
1. User runs multi-agent analysis.
2. If the current result is not yet persisted, save it through the existing `/api/stocks/agents/history` endpoint.
3. User clicks `基于此分析起草交易计划` on the commander card.
4. Frontend calls `POST /api/stocks/plans/draft` with `symbol` and `analysisHistoryId`.
5. Backend returns a draft DTO.
6. Frontend opens an editable modal in `StockInfoTab.vue`.
7. User adjusts numeric/text fields and saves.
8. Frontend calls `POST /api/stocks/plans`.
9. Backend stores a `Pending` plan and auto-adds the symbol to `ActiveWatchlist`.
10. Frontend refreshes a local `planList` section for the current stock page.

### Planned Validation
1. Backend unit tests for draft mapping, nullable numeric extraction, status transition rules, and watchlist integration.
2. SQLCMD verification for the `TradingPlans` table, status column, and `AnalysisHistoryId` column/indexes.
3. Frontend vitest coverage for the commander CTA, draft fetch, editable form fill, and post-save local refresh.
4. If UI code is changed, run Edge/Playwright validation after unit tests.

## ZH
### 摘要
Step 4.2 被规划为一个最小可落地的人机协同交易计划闭环：复用现有 `StockAgentOrchestrator` 已经产出的 commander 归一化结果，由后端基于 `StockAgentAnalysisHistory` 生成正式草稿，前端弹窗允许用户修改，再以 `Pending` 状态保存 `TradingPlan`，并自动把股票加入 `ActiveWatchlist`。

### 当前可复用基础
- `frontend/src/modules/stocks/StockAgentPanels.vue` 已经展示 commander 的摘要、核心判断、触发条件、失效条件、风险上限，非常适合直接增加“起草交易计划”按钮。
- `frontend/src/modules/stocks/StockInfoTab.vue` 已经管理股票 workspace 状态、`selectedAgentHistoryId`，并有现成弹窗模式，可以直接承接 Step 4.2 表单。
- `backend/SimplerJiangAiAgent.Api/Modules/Stocks/StocksModule.cs` 已经有 agent history 与 watchlist 的 minimal API 风格，Step 4.2 应沿用相同模式扩展计划端点。
- `backend/SimplerJiangAiAgent.Api/Data/Entities/StockAgentAnalysisHistory.cs` 已经保存多 Agent 历史，是计划草稿的后端真源。
- `backend/SimplerJiangAiAgent.Api/Infrastructure/Jobs/IActiveWatchlistService.cs` 已提供保存计划后自动进白名单的集成点。

### 还缺的功能
1. 缺 `TradingPlan` 实体、枚举、DbSet、migration、schema initializer 更新。
2. 缺 `ITradingPlanService`，无法做计划的创建、更新、查询、取消。
3. 缺 `ITradingPlanDraftService`，无法根据历史分析记录生成确定性草稿。
4. 缺计划 DTO 和 `/api/stocks/plans*` 接口。
5. 缺 `StockAgentPanels.vue` 的起草按钮。
6. 缺 `StockInfoTab.vue` 的计划弹窗与当前计划区块。
7. 缺后端/前端测试闭环，尤其是“保存计划自动进白名单”。

### 后端设计建议
- `TradingPlan` 建议字段：`Symbol`、`Name`、`Direction`、`Status`、`TriggerPrice`、`InvalidPrice`、`StopLossPrice`、`ExpectedCatalyst`、`InvalidConditions`、`RiskLimits`、`AnalysisSummary`、`AnalysisHistoryId`、`SourceAgent`、`UserNote`、时间戳等。
- 草稿生成禁止再次调用 LLM，必须直接映射 commander 归一化字段：
  - `AnalysisSummary` <- `analysis_opinion`，空时退回 `summary`
  - `ExpectedCatalyst` <- `trigger_conditions`
  - `InvalidConditions` <- `invalid_conditions`
  - `RiskLimits` <- `riskLimits[]` 汇总，空时退回 `risk_warning`
  - `TriggerPrice` / `InvalidPrice` 仅在存在确定性数值时填充，否则返回 `null`
- 创建计划后必须调用 `IActiveWatchlistService.UpsertAsync(symbol, name, "trading-plan", $"plan:{id}", true)`。

### 前端流程建议
1. 用户先完成多 Agent 分析。
2. 如果当前分析还未保存历史，先走现有 `/api/stocks/agents/history` 持久化拿到 `historyId`。
3. 用户点击 commander 卡片下的“基于此分析起草交易计划”。
4. 前端调用 `POST /api/stocks/plans/draft`，传 `symbol` 和 `analysisHistoryId`。
5. 后端返回计划草稿 DTO。
6. `StockInfoTab.vue` 打开可编辑弹窗。
7. 用户调整价格和说明字段后保存。
8. 前端调用 `POST /api/stocks/plans`。
9. 后端保存 `Pending` 计划，并自动写入 `ActiveWatchlist`。
10. 前端刷新当前股票页的 `planList`，立刻显示新计划。

### 计划验证
1. 后端单测覆盖 draft 映射、空价格、状态修改约束、自动进白名单。
2. 用 `sqlcmd` 校验 `TradingPlans` 表、状态字段和 `AnalysisHistoryId` 结构。
3. 前端 vitest 覆盖 commander 按钮、draft 请求、表单填充、保存后刷新。
