# GOAL-008 Step 4.4 Planning Report (2026-03-15)

## EN
### Summary
Planned Step 4.4 as a separate semantic-review layer on top of the existing Step 4.3 deterministic trigger engine. The design keeps all high-frequency rule transitions in the system-layer C# loop, while moving breaking-news threat assessment into an independent worker that can safely degrade when the LLM is unavailable.

### Goal
Use recent local stock news for enabled `ActiveWatchlist` symbols to review whether any `Pending` trading plan is materially threatened by new information that conflicts with the plan's invalidation assumptions.

### Inputs
1. `Pending` `TradingPlan` rows.
2. Enabled `ActiveWatchlist` symbols.
3. Recent `LocalStockNews` items for those symbols.
4. Existing `ILlmService.ChatAsync(...)` abstraction.
5. Existing `TradingPlanEvents` audit surface and frontend short-poll consumption.

### Output
1. Structured review result with `isPlanThreatened`, `reason`, `confidence`.
2. `ReviewRequired` status for threatened plans.
3. Event persistence for both safe reviews and threat reviews.
4. Manual `resume` action to move `ReviewRequired -> Pending` after human confirmation.
5. Frontend rendering of review-required status, linked news context, and review reason.

### Design Decisions
- Keep Step 4.4 out of the Step 4.3 hot path.
- Add a dedicated `TradingPlanReviewWorker` and `TradingPlanReviewService`.
- Reuse `TradingPlanEvents.MetadataJson` for structured traceability/idempotency instead of adding a new review table.
- Limit evaluation to `Pending` plans within enabled `ActiveWatchlist` symbols.
- Only change state when the structured review is threatening and confidence reaches the configured threshold.
- Degrade safely on any LLM/provider/parsing failure: skip semantic review, do not block the main deterministic engine.

### Validation Plan
1. Backend targeted tests for:
   - threatening review -> `ReviewRequired`
   - non-threatening review -> `NewsReviewed`
   - LLM failure -> no status/event mutation
   - per-plan per-news deduplication
   - manual resume -> `ReviewCleared`
2. Frontend unit test for review metadata/status helpers.
3. Frontend production build.
4. SQLCMD schema checks for touched trading-plan columns.
5. Browser MCP validation on the backend-served page:
   - render `待复核`
   - show review detail
   - click `恢复观察`
   - confirm no console/backend runtime errors

## ZH
### 摘要
Step 4.4 规划为建立在 Step 4.3 之上的独立语义复核层，而不是把 LLM 混进盘中高频规则主链。确定性触发仍由系统层 C# 循环负责；突发新闻是否威胁原计划，则由独立 worker 异步判定，并在模型失败时安全降级。

### 目标
利用 `ActiveWatchlist` 白名单内股票的本地个股快讯，对当前 `Pending` 交易计划做“是否已被新信息显著威胁”的动态定性复核，重点比对计划的 `InvalidConditions` 与风险前提。

### 输入
1. `Pending` 状态的 `TradingPlan`。
2. 已启用的 `ActiveWatchlist` 标的。
3. 这些标的最近的 `LocalStockNews`。
4. 现有 `ILlmService.ChatAsync(...)` 抽象。
5. 现有 `TradingPlanEvents` 审计链路与前端短轮询展示面。

### 输出
1. 结构化复核结果：`isPlanThreatened`、`reason`、`confidence`。
2. 对被威胁计划设置 `ReviewRequired`。
3. 对安全复核和威胁复核都落库事件。
4. 提供人工确认后的 `resume` 恢复观察动作。
5. 前端清晰展示待复核状态、关联新闻与复核原因。

### 设计决策
- Step 4.4 不进入 Step 4.3 高频主链。
- 新增独立 `TradingPlanReviewWorker` 与 `TradingPlanReviewService`。
- 使用 `TradingPlanEvents.MetadataJson` 保存结构化追踪/幂等信息，不新增专门 review 表。
- 只评估已启用白名单内的 `Pending` 计划。
- 只有当结构化结果判定为威胁且置信度达到阈值时，才改成 `ReviewRequired`。
- 任意 LLM/provider/解析失败都必须安全降级：跳过语义复核，不影响主规则链。

### 验证计划
1. 后端定向单测覆盖：
   - 威胁复核 -> `ReviewRequired`
   - 非威胁复核 -> `NewsReviewed`
   - LLM 失败 -> 不改状态/不落事件
   - 同计划同新闻去重
   - 手动恢复 -> `ReviewCleared`
2. 前端 review metadata/status helper 单测。
3. 前端生产 build。
4. 用 SQLCMD 检查本轮触及的交易计划相关列。
5. 后端托管页面 Browser MCP 验证：
   - 渲染 `待复核`
   - 显示复核详情
   - 点击 `恢复观察`
   - 检查前端控制台与后端日志无运行时错误
