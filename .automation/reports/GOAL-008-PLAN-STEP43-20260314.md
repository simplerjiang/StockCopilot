### Step 4.3 Goal
Implement the system-layer intraday rule engine that consumes local cache only and deterministically changes a plan from `Pending` to `Triggered` or `Invalid` when price rules are met.

### Inputs
1. Existing `TradingPlan` rows from Step 4.2.
2. Existing `ActiveWatchlist` and `HighFrequencyQuoteService` from Step 4.1.
3. Existing local cache tables:
   - `StockQuoteSnapshots`
   - `MinuteLinePoints`
   - `IntradayMessages`
4. Existing frontend polling patterns in `StockInfoTab.vue`.

### Output
1. `TradingPlanTriggerService` background worker or equivalent service entry.
2. Deterministic status transitions:
   - `Pending -> Triggered`
   - `Pending -> Invalid`
3. Plan event / alert persistence for auditability.
4. Read APIs for plan events / alerts and refreshed plan status.
5. Frontend short-polling consumption of alerts and status changes.

### Scope Boundaries
Step 4.3 must include:
- deterministic price-rule evaluation
- plan status timestamps and event persistence
- idempotent state transitions under repeated polling
- short-polling based frontend status/alert refresh
- optional warning-only volume divergence tagging based on recent minute data

Step 4.3 must not include:
- LLM or semantic review logic
- Step 4.4 news-threat review flow
- SignalR / WebSocket infrastructure
- automatic trade execution or broker integration

### Planned Backend Design
1. Add `TradingPlanTriggerService` that runs on a schedule and reads only local database/cache state.
2. Query only `Pending` plans and join against the latest available quote snapshot for the same symbol.
3. V1 transition rules:
   - for long plans, if `InvalidPrice` exists and latest price `<= InvalidPrice`, mark `Invalid`
   - otherwise, if `TriggerPrice` exists and latest price `>= TriggerPrice`, mark `Triggered`
   - if both conditions are true in the same evaluation, `Invalid` wins
4. Update the plan row with:
   - `Status`
   - `UpdatedAt`
   - `TriggeredAt` or `InvalidatedAt`
5. Add a new event entity for traceability. Suggested fields:
   - `Id`
   - `PlanId`
   - `Symbol`
   - `EventType`
   - `Severity`
   - `Message`
   - `SnapshotPrice`
   - `OccurredAt`
   - `TagsJson` or equivalent optional metadata
6. Persist one event per state change so later reviews can explain exactly why a plan changed.
7. Add warning-only divergence checks from the most recent 30 minutes of minute data. These warnings must not directly flip plan state in Step 4.3.

### Planned API Surface
- `GET /api/stocks/plans?symbol=...` continues to provide current plan state.
- Add an event/alert read endpoint, for example:
  - `GET /api/stocks/plans/alerts?symbol=...&take=...`
  - or `GET /api/stocks/plans/{id}/events`
- API should support both current-symbol view and a compact global board refresh path.

### Planned Frontend Design
1. Reuse existing short-polling patterns in `StockInfoTab.vue` instead of adding SignalR.
2. Refresh current plan state and alert summary on a timer only when a symbol is active.
3. Add a minimal alert presentation:
   - current stock plan card shows latest event summary
   - global board can show latest status/event badge
4. Keep the UI additive; do not redesign the Step 4.2 modal or chart layout in this task.

### Validation Plan
1. Backend targeted tests for:
   - trigger transition
   - invalid transition
   - invalid beats triggered when both conditions match
   - repeated worker passes do not duplicate status events
   - divergence warning creates a warning event without changing plan state
2. Frontend targeted tests for:

3. `npm --prefix frontend run build`
4. Browser validation on the backend-served frontend:

### Step 4.3 目标
实现系统层的盘中规则引擎，只消费本地缓存，通过确定性价格规则把计划从 `Pending` 切到 `Triggered` 或 `Invalid`。

### 输入
1. Step 4.2 已落库的 `TradingPlan`。
2. Step 4.1 已完成的 `ActiveWatchlist` 与 `HighFrequencyQuoteService`。
3. 已存在的本地缓存表：
   - `StockQuoteSnapshots`
   - `MinuteLinePoints`
   - `IntradayMessages`
4. `StockInfoTab.vue` 现有的短轮询刷新模式。

### 输出
1. `TradingPlanTriggerService` 后台服务或等价服务入口。
2. 确定性状态迁移：
   - `Pending -> Triggered`
   - `Pending -> Invalid`
3. 计划事件/告警持久化，满足审计追踪。
4. 计划事件/告警查询接口。
5. 前端基于短轮询消费状态变化与告警。

### 后端设计建议
1. 新增 `TradingPlanTriggerService`，按固定周期运行，但只读本地数据库/缓存，不主动发外部行情请求。
2. 只处理 `Pending` 状态计划，并关联同标的最新 quote snapshot。
3. V1 状态机规则：
   - 做多计划中，若 `InvalidPrice` 有值且最新价 `<= InvalidPrice`，标记为 `Invalid`
   - 否则，若 `TriggerPrice` 有值且最新价 `>= TriggerPrice`，标记为 `Triggered`
   - 同一轮里两者同时命中时，`Invalid` 优先
4. 更新计划记录时同步写：
   - `Status`
   - `UpdatedAt`
   - `TriggeredAt` 或 `InvalidatedAt`
5. 为了审计可追溯，新增计划事件实体，建议字段：
   - `Id`
   - `PlanId`
   - `Symbol`
   - `EventType`
   - `Severity`
   - `Message`
   - `SnapshotPrice`
   - `OccurredAt`
   - `TagsJson` 或等价扩展元数据
6. 每次状态变化必须落一条事件，保证后续能解释“为什么计划变了”。
7. 最近 30 分钟量价背离只作为 `Warning` 级事件，不在 Step 4.3 直接改计划状态。

### 接口建议
