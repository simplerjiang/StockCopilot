# GOAL-008 Step 4.1 Planning Report (2026-03-14)

## EN
### Summary
Planned Step 4.1 as the backend-only market-hours watchlist foundation for the trading-plan engine. The goal is to create a deterministic `ActiveWatchlist` source of truth and a dedicated high-frequency quote worker that refreshes tracked symbols without touching the broader Step 4.2+ trading-plan flow yet.

### Planned Scope
- Add an `ActiveWatchlist` entity/table with timestamps and sync metadata so the system has a stable, queryable whitelist of symbols under active observation.
- Expose minimal backend management endpoints for listing, adding/upserting, touching, and removing watchlist symbols so Step 4.1 can be validated without waiting for the Step 4.2 UI.
- Implement `HighFrequencyQuoteService` as a hosted background worker that only runs during China A-share trading sessions and refreshes quote/minute/message data for enabled watchlist symbols.
- Reuse the existing persistence path into `StockQuoteSnapshots`, `MinuteLinePoints`, and `IntradayMessages` so later trigger services can consume the same local cache tables.
- Add backend unit coverage for market-hours gating, whitelist upsert semantics, and realtime persistence behavior.

### Planned Validation
1. `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj --filter "FullyQualifiedName~ActiveWatchlist|FullyQualifiedName~HighFrequencyQuoteService|FullyQualifiedName~StockSyncService"`
2. `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj`
3. SQL schema verification with `sqlcmd` for the new `ActiveWatchlists` table and indexes.

### Risks Noted
- Background workers in this repo must treat cancellation as normal shutdown, otherwise runtime validation can be polluted by false fatal errors.
- `StockDataService.GetIntradayMessagesAsync` currently prioritizes local facts, so the high-frequency worker should fetch/persist realtime quote payloads directly instead of routing through the local-fact path.
- The project uses both EF migrations and schema initializers; Step 4.1 must update both so fresh DBs and existing local DBs stay aligned.

## ZH
### 摘要
Step 4.1 被规划为交易计划引擎的后端基础设施切片：先建立一个稳定可控的 `ActiveWatchlist` 高频白名单，以及仅在中国 A 股交易时段运行的独立行情 worker，不提前混入 Step 4.2 之后的交易计划 UI 和触发逻辑。

### 规划范围
- 新增 `ActiveWatchlist` 实体/表，带时间戳与同步元数据，让系统有一份可查询、可复用的“当前重点盯盘标的”真源。
- 提供最小后端管理接口，支持查询、新增/更新、触达、删除白名单标的，保证 Step 4.1 不依赖 Step 4.2 的前端表单也能独立验证。
- 实现 `HighFrequencyQuoteService` 后台服务，仅在中国 A 股交易时段轮询启用中的白名单股票，刷新 quote/minute/messages。
- 复用已有的 `StockQuoteSnapshots`、`MinuteLinePoints`、`IntradayMessages` 本地缓存表作为落库目标，为后续触发服务直接提供同一套缓存底座。
- 补充后端单测，覆盖交易时段门控、白名单 upsert 语义和实时落库行为。

### 计划校验
1. `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj --filter "FullyQualifiedName~ActiveWatchlist|FullyQualifiedName~HighFrequencyQuoteService|FullyQualifiedName~StockSyncService"`
2. `dotnet test backend/SimplerJiangAiAgent.Api.Tests/SimplerJiangAiAgent.Api.Tests.csproj`
3. 使用 `sqlcmd` 校验新增 `ActiveWatchlists` 表及索引结构。

### 风险记录
- 本仓库的后台 worker 关闭时必须把取消视为正常退出，否则运行验证会被伪 fatal 错误污染。
- `StockDataService.GetIntradayMessagesAsync` 会优先走本地 facts，因此高频 worker 不能复用它来抓实时消息，而应直接走实时抓取再落库。
- 当前项目同时依赖 EF migration 与 schema initializer，Step 4.1 必须两边同时更新，才能兼容全新数据库和已有本地库。
