# GOAL-016-R6 Plan / 单宿主单进程 packaged runtime 详细设计

## Plan (EN)
- Objective: evolve the current packaged desktop delivery from `desktop EXE + separate packaged backend process` into a single-host, single-process desktop application where one main EXE owns the full lifecycle.
- Hard requirements:
  - users and testers do not need to install any SDK or .NET runtime in advance
  - one main EXE controls both startup and shutdown
  - the backend must no longer survive as a separate long-running background process outside the desktop host
- Accepted boundary:
  - do not optimize for a literally single-file disk artifact
  - allow application-managed companion files, static assets, native dependencies, and a bundled Fixed Version WebView2 Runtime if needed
- Recommended implementation direction:
  - keep the existing localhost + WebView2 frontend contract
  - move ASP.NET Core hosting in-process into the WinForms desktop host
  - treat the desktop EXE as the only user-facing entry point and lifecycle owner

## 计划（ZH）
- 目标：把当前打包版的“桌面 EXE + 独立 packaged 后端进程”交付形态，升级成真正的单宿主、单进程桌面应用，由一个主 EXE 统一控制整个应用生命周期。
- 硬要求：
  - 用户和测试者都不需要预先安装任何 SDK 或 .NET runtime
  - 启动和关闭都由一个主 EXE 统一控制
  - 后端不能再以独立长期后台进程的形式存在于桌面宿主之外
- 已接受的边界：
  - 不再把“磁盘上绝对只有一个文件”作为目标优化方向
  - 允许应用携带自己管理的附属文件、前端静态资源、原生依赖，以及必要时随包发布的 Fixed Version WebView2 Runtime
- 建议方向：
  - 保留现有 localhost + WebView2 前端访问契约
  - 把 ASP.NET Core 后端托管搬进 WinForms 桌面宿主进程内
  - 让桌面 EXE 成为唯一用户入口和生命周期拥有者

## Current Baseline (EN)
- Today the desktop package is not single-process:
  - `desktop/SimplerJiangAiAgent.Desktop/Form1.cs` starts the packaged backend with `Process.Start(...)`
  - `scripts/publish-windows-package.ps1` publishes the backend into `artifacts/windows-package/Backend`
  - closing the desktop window currently kills a separately created backend process rather than stopping an in-process host
- The package is already self-contained for .NET, so the .NET runtime requirement is mostly satisfied today.
- The remaining runtime dependency question is WebView2.

## 当前基线（ZH）
- 当前桌面打包版还不是单进程：
  - `desktop/SimplerJiangAiAgent.Desktop/Form1.cs` 里仍通过 `Process.Start(...)` 拉起 packaged backend
  - `scripts/publish-windows-package.ps1` 仍会把后端单独发布到 `artifacts/windows-package/Backend`
  - 当前关闭窗口时，本质上是 Kill 一个独立创建的后端子进程，而不是停止同进程 host
- 现有包已经是 self-contained，对 .NET runtime 的免安装要求基本已经满足。
- 剩下真正需要明确的运行时依赖是 WebView2。

## Proposed Architecture (EN)
- Host shape:
  - WinForms desktop process remains the shell and UI owner
  - ASP.NET Core host moves in-process and is started/stopped by the desktop app
  - hosted background services remain inside the same process and follow host lifetime
- Frontend shape:
  - keep serving prebuilt Vue static assets from ASP.NET Core
  - keep localhost navigation in WebView2 to minimize frontend churn
- Packaging shape:
  - desktop publish becomes the only runnable application entry
  - backend assemblies become implementation dependencies of the desktop publish, not a separately runnable backend directory
  - companion files remain acceptable for frontend assets, native libraries, and fixed WebView2 runtime payloads

## 建议架构（ZH）
- 宿主形态：
  - WinForms 桌面进程继续作为壳层和 UI 持有者
  - ASP.NET Core host 搬到同一进程内，由桌面程序负责启动和停止
  - 各类 hosted background service 继续保留，但统一跟随同一个 host lifetime
- 前端形态：
  - 继续由 ASP.NET Core 提供预构建 Vue 静态资源
  - 继续保留 WebView2 访问 localhost 的方式，尽量减少前端改动面
- 打包形态：
  - 桌面 publish 成为唯一可运行应用入口
  - 后端程序集变成桌面 publish 的实现依赖，而不再是单独可运行的 Backend 目录
  - 对前端资源、原生依赖和 Fixed Version WebView2 Runtime 等附属文件保持开放，不强行追求单文件

## Implementation Slices (EN)
- Slice 1: backend host extraction
  - refactor the current top-level backend startup into a reusable host builder/runtime service
  - make configuration, content root, static file root, and writable data root injectable from the desktop host
- Slice 2: desktop in-process hosting
  - replace `Process.Start` packaged-backend launch with direct host startup in the desktop process
  - replace process kill-on-close with structured `StopAsync` and host disposal
  - keep health probing, but probe the in-process host startup result rather than an external process lifecycle
- Slice 3: packaging and installer reshape
  - rewrite `publish-windows-package.ps1` so the desktop app is the only runnable output entry
  - stop emitting a separately runnable packaged backend process shape
  - define where frontend dist, default settings, native dependencies, and optional fixed WebView2 runtime land
- Slice 4: WebView2 runtime policy
  - decide between system-installed WebView2 requirement vs bundled Fixed Version Runtime
  - for the stated requirement, bundled Fixed Version Runtime is the safer target
  - add installer/portable validation, upgrade behavior, and rollback notes for that runtime payload

## 实施切片（ZH）
- 切片 1：后端宿主提取
  - 把当前顶层后端启动逻辑重构成可复用的 host builder/runtime service
  - 让配置、内容根目录、静态资源根目录和可写数据目录都可由桌面宿主注入
- 切片 2：桌面进程内托管
  - 用桌面进程内直接启动 host 取代当前 `Process.Start` 拉起 packaged backend
  - 用结构化 `StopAsync` 与 host disposal 取代窗口关闭时的进程强杀
  - 健康检查仍保留，但改为验证同进程 host 的启动结果，而不是外部子进程生命周期
- 切片 3：打包与安装器重构
  - 重写 `publish-windows-package.ps1`，让桌面程序成为唯一可运行输出入口
  - 不再产出独立可运行的 packaged backend 进程形态
  - 明确前端 dist、默认配置、原生依赖和可选 Fixed Version WebView2 Runtime 的落位方式
- 切片 4：WebView2 runtime 策略
  - 在“依赖系统预装 WebView2”与“随包发布 Fixed Version Runtime”之间做明确决策
  - 按本需求，随包发布 Fixed Version Runtime 是更稳妥的目标
  - 为该 runtime 补齐安装器/便携包验证、升级行为和回滚说明

## Cost Assessment (EN)
- Minimal viable version for the stated direction:
  - 2 to 4 working days
  - covers backend host extraction, in-process startup/shutdown, and packaging script reshape
- Full delivery aligned with the stated requirement set:
  - 5 to 8 working days
  - adds fixed WebView2 runtime packaging, installer/portable validation, and upgrade/rollback coverage
- Not recommended scope:
  - do not prioritize a truly single-file disk artifact, because it adds cost while conflicting with realistic WebView2 runtime distribution constraints

## 成本评估（ZH）
- 按当前方向的最小可用版：
  - 预计 2 到 4 个工作日
  - 覆盖后端宿主提取、同进程启动/关闭、打包脚本重构
- 完整满足本次要求集的交付版：
  - 预计 5 到 8 个工作日
  - 额外包括 Fixed Version WebView2 Runtime 打包、安装器/便携包验收，以及升级/回滚链路收口
- 不建议作为当前目标的范围：
  - 不优先追求“磁盘上真正只有一个文件”的绝对单文件交付，因为它会显著抬高成本，并且与 WebView2 runtime 的现实发布约束冲突

## Risks (EN)
- The highest engineering risk is not ASP.NET Core itself, but refactoring the backend startup boundary out of the top-level program safely.
- Hosted background workers must remain non-blocking for the UI thread and must stop cleanly on app exit.
- WebView2 runtime strategy is the real blocker for the “no extra runtime installation” promise.
- Packaging and upgrade size will increase if the fixed runtime is bundled.

## 风险（ZH）
- 最大工程风险不是 ASP.NET Core 本身，而是要把当前顶层程序安全拆成可被桌面宿主复用的启动边界。
- 各类 hosted background worker 需要保证不阻塞 UI 线程，并能在应用退出时干净停止。
- WebView2 runtime 策略才是“用户无需额外安装 runtime”承诺里的真正阻塞点。
- 如果随包带 Fixed Version Runtime，安装包和升级包体积都会增加。

## Development Handoff (EN)

- Required deliverables:
  - updated backend host abstraction
  - updated desktop host lifecycle implementation
  - updated packaging/installer scripts
  - validation evidence for clean-machine startup, clean shutdown, and no leftover backend process

  - 干净机器启动、正常关闭、无残留独立后端进程的验证证据
