# GOAL-009 R2/R3 Planning Report (2026-03-15)

## EN
### Summary
Planned the next two GOAL-009 slices after R1. R2 upgrades the current point-in-time sentiment board into a trend-aware rotation console with 5/10/20-day continuity, diffusion, rank-shift, and stronger stage classification. R3 then pushes those market-stage outputs back into trading-plan and position-guidance flows, while keeping all high-frequency execution paths deterministic and local-first.

### Current R1 Baseline
- Persisted snapshot tables already exist: `MarketSentimentSnapshots`, `SectorRotationSnapshots`, `SectorRotationLeaderSnapshots`.
- Backend already serves `/api/market/sentiment/latest`, `/api/market/sentiment/history`, `/api/market/sectors`, `/api/market/sectors/{sectorCode}`, and `/api/market/sectors/{sectorCode}/leaders`.
- Frontend already has the top-level `情绪轮动` tab with summary cards, paged sector list, and detail panel.
- R1 already fixed three stability constraints that R2 must keep: previous-trading-day fallback, market-turnover denominator, and per-source degradation on Eastmoney failures.

### R2 Goal
Turn GOAL-009 from a single-snapshot board into a rolling market-state console that can answer:
1. Which sectors are only hot today versus persistently strong over 5/10/20 trading days?
2. Whether strength is concentrated in a few leaders or diffusing through the sector breadth.
3. Whether the whole market is in `主升 / 分歧 / 退潮 / 混沌` with higher confidence, not only from one intraday snapshot.

### R2 Data / Schema Plan
Prefer extending the existing snapshot tables instead of creating another family of rollup tables.

1. Extend `MarketSentimentSnapshots` with derived rolling fields:
   - `DiffusionScore`
   - `ContinuationScore`
   - `StageLabelV2`
   - `StageConfidence`
   - `Top3SectorTurnoverShare5dAvg`
   - `Top10SectorTurnoverShare5dAvg`
   - `LimitUpCount5dAvg`
   - `BrokenBoardRate5dAvg`
2. Extend `SectorRotationSnapshots` with rolling and diffusion fields:
   - `RankChange5d`
   - `RankChange10d`
   - `RankChange20d`
   - `StrengthAvg5d`
   - `StrengthAvg10d`
   - `StrengthAvg20d`
   - `DiffusionRate`
   - `AdvancerCount`
   - `DeclinerCount`
   - `FlatMemberCount`
   - `LimitUpMemberCount`
   - `LeaderStabilityScore`
   - `MainlineScore`
   - `IsMainline`
3. Keep `SectorRotationLeaderSnapshots` as the visible ladder table; do not create a full member snapshot table unless R2 implementation proves that breadth aggregates cannot be computed reliably from ingestion-time member scans.
4. If stock-to-sector code mapping is not deterministic enough for R3, add a small dedicated mapping table only after an implementation spike proves the existing stock detail payload is unstable.

### R2 Ingestion / Calculation Plan
1. Reuse `SectorRotationWorker` and `SectorRotationIngestionService`; do not introduce a parallel worker.
2. At each sync:
   - fetch board rankings as today;
   - fetch enough sector constituents to compute breadth aggregates, not only leader ladders;
   - after raw snapshot persistence, compute rolling 5/10/20-trading-day derived metrics from the latest daily snapshot per trading day;
   - persist derived metrics back into the latest snapshot rows of the same sync batch.
3. Promote `盘后` snapshots after 15:05 as the canonical daily bucket for rolling calculations. If a daily close snapshot is missing, use the latest snapshot of that trading day as fallback.
4. Keep all R2 calculations deterministic C# logic. No LLM is allowed in the metric calculator or stage classifier.

### R2 API Plan
1. Extend `GET /api/market/sentiment/latest` to include `DiffusionScore`, `ContinuationScore`, `StageConfidence`, and 5-day averages.
2. Extend `GET /api/market/sectors` to include:
   - `RankChange5d/10d/20d`
   - `StrengthAvg5d/10d/20d`
   - `DiffusionRate`
   - `IsMainline`
   - `MainlineScore`
3. Add `GET /api/market/sectors/{sectorCode}/trend?window=20d` for detailed rolling trend lines and rank shifts.
4. Add `GET /api/market/mainline?window=10d` for a compact current-mainline board list ordered by `MainlineScore`.

### R2 Frontend Plan
1. Keep the existing `情绪轮动` tab as the single page; do not create a separate route.
2. Add a 5/10/20-day compare switch at the top of the page.
3. Add columns or chips to the sector list for:
   - `5日持续`
   - `10日持续`
   - `20日趋势`
   - `扩散度`

   - breadth breakdown (`上涨数 / 下跌数 / 涨停成员数`)
   - leader stability summary
5. Keep the UI stable on missing rolling data by showing neutral placeholders rather than removing the sector row.

### R3 Goal
Use the market-stage and sector-rotation results from GOAL-009 to guide trading-plan pacing and position guidance without turning the system into an automatic trader.

### R3 Integration Principles
1. Market-stage context is advisory and deterministic, not a hard auto-trading switch.
2. R3 may influence displayed position caps, execution frequency, and warnings, but it must not let LLM or market-stage labels override the user-confirmed plan silently.
3. Step 4.3 and Step 4.4 hot paths stay deterministic; GOAL-009 context may enrich decisions or UI, but must not introduce remote dependencies.
4. When no valid market snapshot exists, all R3 outputs must degrade to neutral.

### R3 Backend Plan
1. Extend `/api/stocks/plans/draft` response with a deterministic `marketContext` block:
   - `stageLabel`
   - `stageConfidence`
   - `mainlineSectorName`
   - `mainlineScore`
   - `suggestedPositionScale`
   - `executionFrequencyLabel`
   - `counterTrendWarning`
2. Extend `/api/stocks/position-guidance` to apply a market-stage multiplier from local GOAL-009 snapshots.
3. Persist the market context that existed at plan creation time on `TradingPlan` for auditability:
   - `MarketStageLabelAtCreation`
   - `StageConfidenceAtCreation`
   - `SuggestedPositionScale`
   - `ExecutionFrequencyLabel`
   - `MainlineSectorName`
   - `MainlineScoreAtCreation`
4. If symbol-to-sector matching is deterministic enough, also persist `SectorCodeAtCreation`; otherwise keep only the normalized sector name until a later mapping table is justified.
5. Add a read helper that maps latest GOAL-009 context into watchlist and trading-plan list surfaces.

### R3 Frontend Plan
1. In `StockInfoTab.vue` plan draft modal, show a compact market-context box before save:
   - current stage
   - suggested pace
   - suggested position scale
   - whether the stock is aligned with the mainline sector
2. In current-plan and plan-overview cards, show:
   - stage badge at creation time
   - current stage badge
   - whether the plan is now counter-trend
3. In position-guidance UI, expose how much of the recommendation comes from GOAL-009 market stage versus stock-specific signals.
4. Do not hide or auto-edit user-entered trigger/invalid prices based on GOAL-009 outputs.

### R3 Validation Plan
1. Backend unit tests:
   - market-stage to position-scale mapping
   - neutral fallback when no snapshot exists
   - plan draft enrichment uses local snapshot data only
2. Frontend unit tests:
   - market-context card in draft modal
   - current-plan / overview badges
   - neutral fallback copy
3. Browser MCP:
   - generate plan draft
   - verify market-context box renders
   - verify saved plan cards show stage context
   - verify no console/runtime errors

## ZH
### 摘要
R1 完成后，GOAL-009 的后两段已经规划清楚。R2 的目标是把现在这个“单次快照型”情绪轮动页，升级成真正能看 5/10/20 日持续性、扩散度、排名变化和主线强度的趋势控制台；R3 再把这些阶段判断接回交易计划和仓位建议，但仍然坚持 Local-First 和确定性主链，不把系统变成自动交易器。

### 当前 R1 基线
- 已有三张快照表：`MarketSentimentSnapshots`、`SectorRotationSnapshots`、`SectorRotationLeaderSnapshots`。
- 后端已提供 `/api/market/sentiment/latest`、`/api/market/sentiment/history`、`/api/market/sectors`、`/api/market/sectors/{sectorCode}`、`/api/market/sectors/{sectorCode}/leaders`。
- 前端已有顶层 `情绪轮动` 页签，能展示摘要卡、分页板块榜和详情面板。
- R1 已经补齐三条稳定性底线，R2 必须继承：最近交易日回退、成交集中度按全市场成交额口径、东财单点失败按数据源粒度降级。

### R2 目标
把 GOAL-009 从“当前这一刻谁热”升级成“近 5/10/20 个交易日谁是真的主线、谁只是短热”的趋势控制台，重点回答：
1. 哪些板块只是今天冲高，哪些板块在 5/10/20 日上持续走强。
2. 板块热度是只靠少数龙头撑着，还是已经向更多成分股扩散。
3. 大盘当前到底更像 `主升 / 分歧 / 退潮 / 混沌` 中的哪一种，而且判定置信度更高。

### R2 数据与表结构计划
优先扩展现有快照表，不再新开一整套 rollup 表。

1. 扩展 `MarketSentimentSnapshots`：
   - `DiffusionScore`
   - `ContinuationScore`
   - `StageLabelV2`
   - `StageConfidence`
   - `Top3SectorTurnoverShare5dAvg`
   - `Top10SectorTurnoverShare5dAvg`
   - `LimitUpCount5dAvg`
   - `BrokenBoardRate5dAvg`
2. 扩展 `SectorRotationSnapshots`：
   - `RankChange5d`
   - `RankChange10d`
   - `RankChange20d`
   - `StrengthAvg5d`
   - `StrengthAvg10d`
   - `StrengthAvg20d`
   - `DiffusionRate`
   - `AdvancerCount`
   - `DeclinerCount`
   - `FlatMemberCount`
   - `LimitUpMemberCount`
   - `LeaderStabilityScore`
   - `MainlineScore`
   - `IsMainline`
3. `SectorRotationLeaderSnapshots` 继续作为可视化龙头梯队表保留；只有当实现验证证明“仅靠采集时聚合无法稳定算出扩散度”时，才额外引入成分股快照表。
4. 如果 R3 落地时发现 symbol -> sectorCode 的映射不够稳定，再新增一个很小的映射表；在那之前不要提前加表。

### R2 采集与计算计划
1. 继续复用 `SectorRotationWorker` 和 `SectorRotationIngestionService`，不新增并行 worker。
2. 每次同步时：
   - 先抓板块榜；
   - 再抓足够的板块成分股，用于算扩散度，不只抓龙头前几名；
   - 原始快照落库后，再按“每个交易日最后一份有效快照”回算 5/10/20 日派生指标；
   - 把这些滚动指标回写到本次同步生成的最新快照行。
3. 15:05 之后的 `盘后` 快照作为 R2 日线滚动计算的优先日桶；如果当天缺失 `盘后`，则回退到当天最后一份快照。
4. R2 的所有计算都保持为确定性 C# 逻辑，不允许把 LLM 拉进阶段分类器或扩散度计算器。

### R2 API 计划

2. 扩展 `GET /api/market/sectors`：返回

4. 新增 `GET /api/market/mainline?window=10d`：返回当前主线板块列表，按 `MainlineScore` 排序。

### R3 目标
把 GOAL-009 的阶段判断接回交易计划和仓位建议，但仍然遵守“辅助决策、不自动交易”的边界。

### R3 集成原则
1. 市场阶段只做确定性建议增强，不做静默自动下单或自动改计划。
2. R3 可以影响展示的仓位上限、执行节奏和风险提示，但不能绕过用户已经确认的交易计划。
3. Step 4.3 / 4.4 的高频主链仍然保持确定性；GOAL-009 只允许作为本地快照上下文接入。
4. 若当前没有有效市场快照，R3 一律降级为中性，不允许报错或臆断。
