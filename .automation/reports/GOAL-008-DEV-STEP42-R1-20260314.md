# GOAL-008 Step 4.2 R1 Development Report (2026-03-14)

## EN
### Scope
Completed the post-Step-4.2 trading-plan usability enhancement: editable pending plans, hard delete support, deterministic stop-loss/take-profit/target defaults from saved agent history, and a global cross-stock board in the stock terminal.

### Development
- Backend model and APIs:
  - Extended `TradingPlan` and plan DTOs with `TakeProfitPrice` and `TargetPrice`.
  - Extended `TradingPlanService` with optional global listing and `DeleteAsync(...)`.
  - Extended `StocksModule.cs` so `/api/stocks/plans` can be queried without `symbol` for board mode, and added `DELETE /api/stocks/plans/{id}`.
  - Replaced the default EF string-enum conversion for `TradingPlan.Status` with tolerant parsing so legacy local rows no longer crash the global board when old string values such as `Archived` are present.
- Deterministic draft defaults:
  - Extended `TradingPlanDraftService` to derive stop-loss/take-profit/target values from saved analysis history only.
  - Price precedence is deterministic and numeric-only:
    - commander chart fields first
    - financial `institutionTargetPrice` next
    - trend forecast max price fallback
- Legacy schema compatibility:
  - Patched the schema initializer and the Step 4.2 migration SQL additively to add `TakeProfitPrice` and `TargetPrice` when a legacy local `TradingPlans` table already exists.
- Frontend trading-plan UX:
  - Added a visible `交易计划总览` board that loads without selecting a stock.
  - Added plan edit/delete actions in the current-stock plan list.
  - Reused the modal for create and edit, including `止盈价` and `目标价` fields.
  - Refreshed both the per-stock plan list and the global board after save and delete.
  - Hardened failed-response parsing in `StockInfoTab.vue` so failed fetches fall back cleanly instead of surfacing the browser-level `body stream already read` error.
- Tests:
  - Updated backend `TradingPlanServicesTests` to cover the new price fields and delete behavior.
  - Updated `StockInfoTab.spec.js` to cover board loading, price-field persistence, edit flow, and delete flow.
  - Fixed one existing sidebar refresh test by scoping the selector to the intended card after the new board introduced another `刷新` button.

### Validation
- `dotnet test .\backend\SimplerJiangAiAgent.Api.Tests\SimplerJiangAiAgent.Api.Tests.csproj --filter TradingPlanServicesTests` -> passed (9/9)
- `npm run test:unit -- src/modules/stocks/StockInfoTab.spec.js` from `frontend` -> passed (35/35)
- `npm run build` from `frontend` -> passed
- Browser MCP on backend-served `http://localhost:5119/`:
  - `/api/stocks/plans?take=20` now returns `200` on the legacy local DB.
  - `交易计划总览` renders actual plan cards, including normalized legacy statuses.
  - After selecting `深科技`, `当前交易计划` still shows `编辑` / `删除` and the `止盈` / `目标` fields.

### Issues / Notes
- The first frontend test run exposed a stale selector assumption in an existing news-impact refresh test; the implementation was correct, but the test clicked the new trading-plan board refresh button because both cards now contain a `刷新` action. The fix was to scope the selector to `.news-impact-header button`.
- Browser validation exposed a real local-data compatibility bug after all unit tests had already passed: legacy `TradingPlans.Status` strings (`Draft`, then `Archived`) caused EF enum materialization failures and broke the new board endpoint with `500`. The final fix was to move status parsing to a tolerant converter instead of whitelisting one legacy enum value at a time.
- Local reruns of `start-all.bat` were also producing noisy `address already in use` startup failures on port `5119`; the launcher now checks the health endpoint first, reuses a healthy backend, and only force-stops a prior listener when it belongs to the same app.
- This R1 finishes the editing/deletion/global-board usability gap on top of Step 4.2. GOAL-008 overall remains open for later Step 4.3-4.5 work.

### 问题 / 说明
- 第一次前端单测失败并不是实现错误，而是旧用例仍假设页面里只有一个 `刷新` 按钮；新增总览后，这个假设失效，所以把选择器收窄到 `.news-impact-header button`。
- 浏览器联调证明仅靠单测还不够：虽然单测已全部通过，但本地旧库里历史 `TradingPlans.Status` 字符串（先暴露 `Draft`，再暴露 `Archived`）仍会让 EF 枚举实体化失败，并把总览接口打成 `500`。最终通过把状态读取改为宽容解析，才彻底修复该兼容性问题。
- 本地重复执行 `start-all.bat` 时，固定端口 `5119` 也会打出 `address already in use` 噪声失败；现已把启动脚本改成先查健康、已健康则复用、仅在旧监听确认为同一应用时才回收重启。
- 本次 R1 已补齐 Step 4.2 在编辑、删除、总览可见性上的缺口；GOAL-008 整体仍待后续 Step 4.3-4.5 继续推进。
