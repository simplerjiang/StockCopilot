【问题治理计划（2026-03-02）】

目标：先把问题拆成可落地任务，再按优先级逐项实施、验证、上线。

------------------------------------------------------------
问题 1：资讯不够“新鲜/可靠”（会抓到很旧内容）
------------------------------------------------------------
现象：个股资讯 Agent、板块资讯 Agent 有时返回几个月甚至几年前信息，影响交易判断。

根因假设：
1) Prompt 未强制时间窗口（例如最近 24h / 72h / 7d）。
2) 检索层未执行发布时间过滤与来源白名单。
3) 聚合层缺少“时效分+来源可信度分”排序。
4) 输出层未强制披露“证据时间戳与来源”。

解决方案（按实施顺序）：
1) Prompt 强约束（最低风险，先做）
	- 为资讯相关 Agent 增加硬性规则：
	  - 默认只允许最近 72h 内容；若不足可放宽到 7d，并在结果明确标注“扩窗”。
	  - 禁止使用无发布时间或发布时间不可解析的数据作为核心证据。
	  - 每条结论必须附带来源、发布时间、抓取时间。
2) 检索与聚合增强（核心）
	- 在后端检索层加入时间过滤（可配置：24h/72h/7d）。
	- 引入来源白名单与来源权重（官媒/交易所公告/主流财经媒体 > 低质量站点）。
	- 结果排序改为综合分：recency_score + source_score + relevance_score。
3) 输出结构化校验（兜底）
	- 若关键字段缺失（source/publishedAt/evidence），降级为“信息不足，观望”。
	- 前端展示证据时间，超时证据高亮“过期风险”。

验收标准：
A) 资讯结果中，最近 72h 证据占比 >= 80%。
B) 所有结论都可追溯到来源 + 发布时间。
C) 无发布时间证据不得作为“买入/卖出”核心依据。

测试计划：
- 单元测试：时间窗过滤、来源权重排序、缺失字段降级。
- 回放测试：用近 7 天热点标的验证新鲜度和稳定性。

------------------------------------------------------------
问题 2：分析前后矛盾（涨一天就大利好，跌一天就大利空）
------------------------------------------------------------
现象：模型容易追随单日涨跌切换叙事，结论一致性差。

根因假设：
1) 缺少“多时间尺度一致性”约束（1D 与 5D/20D 冲突未处理）。
2) 缺少状态机（趋势延续/震荡/反转）导致叙事跳变。
3) 缺少上次结论记忆与差异解释（为什么改判）。
4) 缺少置信度与触发/失效条件，导致语义绝对化。

解决方案（按实施顺序）：
1) 结论模板标准化（先做）
	- 强制输出：方向、置信度、触发条件、失效条件、风险上限、证据列表。
	- 禁止“绝对性措辞”，必须给出可验证条件。
2) 一致性约束引擎（核心）
	- 引入多周期信号融合（1D/5D/20D）；若短期与中期冲突，输出“分歧态”。
	- 增加“最小变更原则”：与上次结论差异较大时，必须输出“改判原因”。
	- 新增“历史结论带入”约束：仅对指挥者 Agent 注入近 3-7 天分析记录（默认 5 天，可配置 3/5/7），其他资讯/执行类 Agent 不注入历史结论，避免全链路偏置放大。
	- 历史注入内容最小化：仅带入方向、置信度、触发/失效条件、关键证据摘要、最终结果标签；每次最多 N 条（建议 5-10 条）并按时间倒序。
3) 状态机与滞后机制（稳定器）
	- 定义市场状态：延续、震荡、反转。
	- 单日波动不直接触发方向翻转，需满足阈值或连续确认条件。
	- 状态机读取“近 3-7 天指挥者历史结论”作为先验，但若当日出现强反证（触发失效条件）可覆盖先验并强制输出改判解释。

验收标准：
A) 连续两日同标的分析中，“无解释改判”比例趋近 0。
B) 输出均包含触发/失效条件与风险上限。
C) 短中周期冲突时明确标注“分歧态”，不再强行单边结论。
D) 仅指挥者 Agent 读取近 3-7 天历史结论；其他 Agent 的上下文不包含该历史包。

测试计划：
- 单元测试：多周期融合、一致性检查、改判解释必填。
- 单元测试：历史注入范围校验（仅指挥者 Agent）、3/5/7 天窗口过滤、条数上限与时间倒序。
- 场景回测：上涨日/下跌日交替样本，验证叙事稳定性。

------------------------------------------------------------
实施优先级与节奏（我们将按此顺序逐项改）
------------------------------------------------------------
Phase 1（先解决新鲜度）：问题 1 的 Prompt 强约束 + 时间窗过滤 + 输出证据字段。
Phase 2（再解决矛盾）：问题 2 的结论模板 + 一致性约束 + 改判解释。
Phase 3（联调收敛）：联合回放测试与 Edge 交互验证，必要时调参。

每个 Phase 均执行：开发 -> 单测 ->（涉及前端时）Edge 验证 -> 记录报告。
